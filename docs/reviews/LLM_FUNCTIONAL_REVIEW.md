# Функциональный обзор локальной LLM (gemma) — 2026-06-09

> Запрошен владельцем: «протестируй хорошенько нашу LLM, есть сомнения — развенчай или порекомендуй
> лучше». Прогон против живого `gemma-4-26B-A4B-it-UD-Q4_K_XL` @ `192.168.0.29:8080` (chat) и
> `bge-m3` @ `:8083` (RAG-эмбеддинги). Метод — прямые запросы к `/v1/chat/completions` + чтение кода
> провайдера `ai/chat.rs`.

## TL;DR

**Модель в порядке — отвечает корректно и качественно.** Сомнения объясняются НЕ «плохой моделью», а
тем, что **gemma здесь — reasoning-модель** (chain-of-thought), а приложение под это не настроено:

1. **Главный баг UX:** парсер стрима (`ai/chat.rs`, `struct Delta`) читает только `delta.content` и
   **игнорирует `delta.reasoning_content`**. gemma сначала отдаёт длинный поток `reasoning_content`
   (десятки токенов «размышления»), и лишь потом — `content` с ответом. Итог: после клика
   «Генерирую…»/⌘I — **много секунд мёртвой тишины** (UI ничего не стримит, пока модель «думает»),
   затем разом появляется ответ. Это и есть ощущение «зависло».
2. **На примитивных задачах reasoning вреден.** Замер (inline-rewrite, `max_tokens=250`, temp 0):
   - reasoning **ON**: **6.9с**, 250 токенов — и **ответ ПУСТОЙ** (`finish_reason=length`): рассуждение
     съело весь бюджет токенов, до ответа не дошло.
   - reasoning **OFF**: **3.8с**, 133 токена — нормальный ответ.
   То есть на коротких задачах (inline сократить/переписать, дайджест, судья противоречий) reasoning не
   только медленнее, но при тесном `max_tokens` даёт **пустой/обрезанный** результат.
3. **Reasoning отключается без смены модели.** Из проверенных способов работает **только**
   `chat_template_kwargs: {"enable_thinking": false}` (0 токенов рассуждения, мгновенный ответ).
   НЕ работают: `reasoning_effort:"none"`, `/no_think` в промпте, системная инструкция «без рассуждений».

**Ответ на вопрос «нужна ли быстрая модель для примитива»:** **отдельная модель не обязательна** —
достаточно слать `enable_thinking:false` для примитивных задач на той же gemma (≈2× быстрее, без пустых
ответов). Отдельная маленькая модель дала бы выигрыш в throughput/RAM, но это оптимизация второго шага,
а не необходимость. Главное — починить две вещи ниже.

## Доказательства (прямые прогоны)

| Проверка | Результат |
|---|---|
| `model` | `gemma-4-26B-A4B-it-UD-Q4_K_XL.gguf`, чат отвечает (200 OK) |
| «Столица Франции?» `max_tokens=200` | `content="Париж"`, но `completion_tokens=95` (≈90 — рассуждение) |
| `max_tokens=20` | `content=""` — все 20 токенов ушли в reasoning |
| stream: ключи `delta` | `{role, reasoning_content, content}` — рассуждение в ОТДЕЛЬНОМ поле, идёт ПЕРВЫМ |
| `enable_thinking:false` | reasoning=0, `content="Париж"` сразу ✅ |
| `reasoning_effort:none` / `/no_think` / system-промпт | reasoning НЕ отключился ❌ |
| inline-rewrite ON vs OFF | 6.9с/пусто vs 3.8с/ответ |

Качество ответов (там, где reasoning успевает завершиться) — хорошее: RU/EN, по делу. RAG-эмбеддер
bge-m3 на golden-наборе даёт recall@8=1.0, nDCG@8=0.883 (см. `eval/baseline.json`) — ретрив здоровый.

## Рекомендации (по приоритету)

### R1 — Обрабатывать `reasoning_content` в стриме (обязательно)
`ai/chat.rs`: добавить `reasoning_content: Option<String>` в `Delta`; в `parse_sse_delta` различать
`SseEvent::Content` и `SseEvent::Reasoning`. В `chat_rag`/`inline_complete` слать новый эвент канала
(`ChatStreamEvent::Reasoning { text }` / `InlineStreamEvent::Reasoning`). Фронт показывает их как
сворачиваемый блок «💭 размышляет…» или хотя бы статус-индикатор — **чтобы не было мёртвой тишины**.
Минимальный безопасный шаг (если без фронта): эмитить разовый статус «модель размышляет…» при первом
`reasoning_content`, чтобы UI знал, что процесс идёт.

### R2 — Отключать reasoning для примитивных задач (обязательно)
Inline (continue/rewrite/summarize), дайджест, судья противоречий, будущие умные-шаблоны/авто-теги —
короткие/структурные задачи, где CoT вреден. Прокинуть в `OpenAiChatProvider` режим «без рассуждений»
→ в теле запроса `chat_template_kwargs:{enable_thinking:false}`.
- Реализация: добавить в провайдер флаг/режим (напр. `ChatOptions { reasoning: bool }` как параметр
  `stream_chat`, ИЛИ отдельный конструктор `OpenAiChatProvider::fast()`), и для примитивов строить
  «быстрый» вариант. RAG-чат оставить с reasoning (качество).
- Доп. защита: для судьи противоречий (JSON-вердикт) reasoning особенно опасен — может обернуть JSON в
  рассуждение и сломать `parse_judgment`. Проверить отдельно (TODO ниже).

### R3 — Поднять `max_tokens` для RAG-чата (важно)
Раз reasoning ест бюджет, у чата `max_tokens` должен быть с запасом (иначе длинное рассуждение обрежет
ответ). Проверить текущее значение в `build_*`/провайдере и заложить запас (или динамически).

### R4 — Быстрая модель для примитива (опционально, позже)
Не обязательно (R2 решает скорость), но маленькая non-reasoning модель (напр. отдельный порт) дала бы
ещё меньше латентность/RAM на массовых примитивах (авто-теги, классификатор шаблонов, заголовки). Решать
после R2 по факту нагрузки. Вписывается в текущую инфру (отдельный chat-эндпоинт в `.nexus/local.json`).
