## Сквозные темы

1. Usable из коробки для НЕ-владельца — сквозной разрыв №1, но С ПРАВИЛЬНОЙ последовательностью: «Новая заметка» (0 команд создания, verified) ПОДЛИННО независима → отгружается первой как настоящий quick-win; LLM-settings (0 config-UI) несёт СКРЫТУЮ зависимость — config читается раз в open_vault (vault.rs:71-123), поэтому без hot-apply-пути (rebuild RAG/chat ctx + re-probe dim + условный reconcile) сохранение настроек = тихий no-op до рестарта = ХУЖЕ текущего. Backend готов, но это L-эпик (11a форма + 11b hot-apply), не near-pure-frontend. Невидим прогресс индексации/статус AI.

2. Единая граница доверия для сети ядра как ОДНО composition-root решение: chat/embedder ходят мимо allowlist/audit/opt-in/kill-switch (только redirect=none, verified), git стейджит add_all("*") вопреки §8.2 (утечка СЕГОДНЯ), §4.3 AIClient — фантом (0 в коде). КЛЮЧЕВОЕ: §4.3-док-фикс и egress-хелпер — НЕ два пункта, а одно ADR-решение → ADR/док СНАЧАЛА (дёшево), фасад conform-ит; иначе док сядет как «план», а хелпер построит другую форму = передрейф. Probe из LLM-settings — первый egress-вектор, идёт через guarded_client. git-фикс выше хелпера на risk-now (реальная утечка vs будущие пути), но требует ДИЗАЙНА списка типов (themes/** = произвольный JS/CSS).

3. Путь к distributable заблокирован на двух фронтах + ВНУТРЕННЯЯ связность конфига: внешний (Apple $99/год, Authenticode, Tauri-keypair) — недели, запускать СЕЙЧАС параллельно; автономный (release.yml, версия 0.0.0×4, ядро updater, smoke) — но ПЕРЕД всем нужен RANK-0 de-risk `tauri build --debug` (CI его не запускает → бандлится ли вообще — неизвестно), и updater-конфиг делать ПОСЛЕ подпись-формы bundle-config (hardenedRuntime/createUpdaterArtifacts взаимоперезаписываются). Сегодня 0 артефактов, 0 тегов.

4. Фундамент под vision-класс через примитив миграций как ЗАМОК резюмируемости: нет планировщика джобов (0 в коде, блокирует News Feed/Карту/Противоречия/Home/память-агента), нет rebuild FTS5/usearch в раннере (§5.1 заявлен, кода нет даже хука). rebuild-примитив (п.13) — ЖЁСТКАЯ зависимость chat-persist/re-chunk/jobs-таблицы: если любая схемо-миграция ляжет раньше, первая chunk-миграция ломает .nexus у пользователя. Планировщик требует ADR владельца ДО первой зависимой фичи.

5. Гейты против ложной зелени + непокрытая склейка + git-в-CI: traceability не проверяет существование имён, 11 #[ignore] без whitelist, нет --allowOnly=false, глобальный coverage-floor скрывает слабые модули; command-слой (28 команд, git_* — 10) и E2E (UI→IPC→Rust) не покрыты. ДОБАВЛЕНО критикой: git-integration/selective-staging тесты упираются в git-identity и keychain (creds.rs), которых нет на headless Linux → зелено локально, скип/падение в CI. Дешёвая автоматика + integration-крейт раньше дорогого E2E; типизированные ошибки (AppError) ПЕРЕД крейтом, иначе ассерты против String переписываются.

6. Масштаб 50k — числа это проекции, НО perf-пункты переплетены в единый эпик: per-tick React-рендер графа (кап 600), отсутствие cross-file batching (семафор простаивает — но фикс ЛОМАЕТ инвариант единой задачи, это L не M), F32-usearch (~1GB+ RSS), плейсхолдер-токенайзер (кириллица ×1.5-2 протекает во ВСЕ числа), отсутствие mmap/cache, тёплый бенч без cold-index/RAG-TTFT. Порядок ОБЯЗАТЕЛЕН: tokenizer→batching→quantization+PRAGMA→cold-bench, иначе взаимная инвалидация замеров. Измерительная половина не полностью автономна (живой embedder нестабилен).

7. Документация дезинформирует автономного агента — и УЖЕ отравила это планирование: §2-раскладка/§4.3 AIClient/§5.1 rebuild/config-дефолты/TESTING_STRATEGY §2/стрей-строка расходятся с кодом; фантомный chat_messages из линзы безопасности протёк в синтез (таблицы НЕТ — 001/002/003, греп подтверждает; at-rest чат-контента не существует до chat-persist). AC-Q-6 — отгруженный AC, требующий АВТОМАТ-линта висячих упоминаний, а не разового прохода (передрейфует). Раз дрейф искажает ситуационную модель агента, дешёвый синк-проход + линт едут рядом с гейтами как correctness-процесса, не на дне списка.
