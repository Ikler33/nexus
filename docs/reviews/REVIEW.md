# Архитектурное ревью — Nexus

> Синтез пяти независимых ревью: **R** — Rust/Tauri-инженер · **ML** — ML/RAG-инженер · **P** — Plugin-эксперт · **F** — Frontend/редактор-инженер · **S** — Security-инженер.
> Ссылки на разделы — по нумерации `ARCHITECTURE.md`.

Главный вывод заранее: план детальный и продуманный по структуре, но содержит несколько дефектов **уровня «не скомпилируется / не заработает as-is»** в ядре индексации и RAG, и **внутренне противоречивую плагинную модель**, которая одновременно несовместима сама с собой, небезопасна и не запускает экосистему — а экосистема заявлена как non-negotiable. Это нарушает собственный принцип плана «Correctness over speed».

## Блокеры (исправить до написания кода)

Отсортировано по критичности. Чем больше ревьюеров независимо нашли проблему — тем сильнее сигнал.

### Б1 · Плагинная модель некогерентна: WASM-логика против TypeScript-SDK
**Нашли:** P, R, F, S (4 из 5)
**Проблема:** Раздел 7.3 описывает богатый асинхронный JS-API — `registerEditorExtension(ext)` возвращает живое CodeMirror-расширение, `onFileChanged(cb)` хранит JS-замыкание, `complete()` отдаёт `AsyncIterable<string>`. Но разделы 4.4/7.1/7.4 объявляют логические плагины как `main.wasm` в Wasmtime. WASM-модуль физически не может сконструировать `ViewPlugin`/`Decoration` CodeMirror (они живут в webview, не в Rust-процессе), не может «держать» JS-колбэк, не имеет `Symbol.asyncIterator`. Эти объекты невозможно протащить ни через WASM-границу, ни через `postMessage`/structured-clone. Третий тип — JS-в-iframe — изолирован opaque-origin и тоже не дотянется до главного инстанса CodeMirror. То есть редакторные расширения (80% ценности заметочника) нереализуемы ни в одном из трёх заявленных типов плагинов. Appendix B легкомысленно ставит «editor extensions» в минор v1.2, как будто это инкремент, а не разрыв всей модели.
**Как исправить:** Выбрать ОДНУ модель и переписать раздел 7. Рекомендация (путь Obsidian/VS Code, и его поддержали R/P/F): основной тип — **доверенный JS/TS-плагин** с прямым типизированным доступом к app и CodeMirror, изоляция через отдельный JS-контекст (worker для логики + контролируемый main-thread мост для редакторных расширений). WASM понизить до опционального под-формата для чистых вычислений, вызываемого из JS-плагина; iframe — только для тяжёлого изолированного UI-вью без прямого доступа к CodeMirror. Добавить в 7.6 матрицу «тип плагина × доступные возможности», чтобы никто не обещал inline-виджеты песочнице, которая их не тянет.

### Б2 · Граница безопасности плагинов декоративна: capability laundering + неаутентифицируемый postMessage + слепые permissions
**Нашли:** S, P (2, security-критично)
**Проблема:** Раздел 11/Уровень 3 продаёт `iframe sandbox="allow-scripts"` без `allow-same-origin` как безопасность. Это ложная модель угроз:
- **Capability laundering.** Песочница блокирует прямой `fetch` плагина, но 7.5 проксирует весь vault-API через `postMessage`, и привилегированную операцию выполняет доверенный хост. Плагин с `vault:read` + `ai:complete` при настроенном облачном `fallback_provider` читает все 50k файлов руками хоста и эксфильтрует их в облако легальным вызовом `nexus.ai.complete(<контент>)`. Сеть плагину не нужна.
- **Невозможно аутентифицировать отправителя.** Sandboxed iframe имеет opaque origin `"null"` → проверка `event.origin` бесполезна, все плагины неразличимы. `pluginId` в payload (7.5) задаёт сам плагин и может наврать → confused deputy: плагин A с `vault:read` присылает сообщение с `pluginId:"B"`, у которого `vault:write`. Без привязки сообщения к плагину вся permissions-система не enforce'ится.
- **Permissions слепые.** Install-time consent «как Android» (сам Android выпилил это в 2015), `vault:read` = весь vault включая пароли/секреты в заметках, нет path-scoping, нет отзыва, нет аудита.
**Как исправить:** Перестать продавать sandbox как границу — граница это **host-side capability broker**, его и проектировать. На каждый плагин — отдельный `MessageChannel`, один `MessagePort` через `transfer`; identity по порту, `pluginId` из payload убрать. Path-scoped permissions (`{"vault:read":["Work/**","!Private/**"]}`), runtime-grant для чувствительного, отзыв в UI, обязательный неотключаемый audit-log (что читал/писал, объём egress, AI-вызовы). `ai:complete` плагина при включённом облаке — принудительно `localOnly` или отдельное подтверждение. Развести `ai:complete` (промпт плагина) и `ai:completeWithVaultContext` (контекст подбирает хост).

### Б3 · Git разносит исполняемый код по устройствам, а подпись отложена до v2.0
**Нашли:** S, P (2)
**Проблема:** `.gitignore` (8.1) делает `!.nexus/plugins/` версионируемым → бинарь плагина (WASM/native + UI) коммитится и синхронизируется (8.2 `add_all(["*"])`, `auto_commit:true`) и тянется на все устройства и mobile (8.4) через `auto_pull:true`. При этом plugin signing отложен до v2.0 (11/Уровень 4), а marketplace с community-плагинами едет в Фазе 2 — месяцы между «можно ставить чужой код» и «код проверяется». Итог: один скомпрометированный девайс или одно вредоносное обновление = тихая авто-активация малвари на всех синхронизированных устройствах. Это превращает git-sync заметок в механизм доставки кода.
**Как исправить:** Не версионировать исполняемый код. В git хранить только декларацию `id@version#sha256` + настройки (это и так есть в `config.json.plugins.enabled`), бинарь ставить локально из реестра по хэшу с отдельным consent на каждом устройстве. Приход нового/изменённого плагина через pull → состояние `disabled, needs-review`. Подпись поднять в Фазу 2 (в одну фазу с marketplace). Mobile принудительно игнорирует `.nexus/plugins/`.

### Б4 · Ядро RAG сломано: эмбеддинг всего файла одним вектором вместо чанков
**Нашли:** ML, R (2)
**Проблема:** В `reindex_file` (4.2) код вызывает `self.ai_client.embed(&content)` — один вектор на весь файл, затем `tx.upsert_chunks(path, &parsed.chunks, &embedding)` суёт **один** `embedding` на **массив** чанков. Это прямо противоречит чанкеру (6.1, режет на 512-токенные куски), схеме `vec_chunks` (per-chunk) и даже самому trait'у `LLMProvider::embed(texts: &[String])` (4.3). Либо все чанки файла получают один файловый вектор (гранулярность поиска деградирует до файла, чанкинг бессмысленен), либо код не компилируется по типам. Плюс эмбеддер llama.cpp имеет лимит контекста — файл на 50 КБ молча обрежется. Дополнительно `tokio::join!(parse, embed)` (4.2) ложно параллелит: embed чанков **зависит** от parse, его нельзя считать параллельно.
**Как исправить:** Pipeline `read → blake3 → parse → chunk → batch-embed каждого чанка (под `Semaphore` к llama.cpp) → одна транзакция`. `embed` принимает `&[String]` чанков, `upsert_chunks` пишет `Vec<(chunk, Vec<f32>)>` 1:1. Перед вставкой — удалять старые векторы чанков файла. (SA Agent уже делает именно так: вектор на каждый чанк.)

### Б5 · Размерность вектора захардкожена 1024, а модель сконфигурирована на 768
**Нашли:** ML, R (2)
**Проблема:** Схема (раздел 5): `embedding FLOAT[1024]`. Конфиг (раздел 5): `"embedding_model":"nomic-embed-text"` — это **768** измерений. sqlite-vec фиксирует размерность в DDL при `CREATE VIRTUAL TABLE` и отвергнет/испортит вектор длины 768. Сломается на первом же эмбеддинге. SA Agent уже наступил на это (BGE-large 1024 → nomic 768, старые векторы авто-скипались).
**Как исправить:** Размерность брать в рантайме из ответа эмбеддера; генерировать DDL `vec_chunks` под актуальную модель; хранить `embedding_model`/`embedding_dim`/`embedding_version` в `settings`; при смене любого — полная переэмбеддизация векторного слоя.

### Б6 · sqlite-vec — это brute-force flat scan, а не HNSW/ANN: все SLA производительности фиктивны
**Нашли:** ML, R, F (2–3)
**Проблема:** В трёх местах (раздел 3 «ANN/HNSW индекс», 6.2 «sqlite-vec ANN», 10 «~50ms для 500k чанков») утверждается ANN. Фактически sqlite-vec на текущий момент делает полный линейный перебор; HNSW в нём не реализован. 500k × 768 float32 ≈ 1.5 ГБ, читаемые и скалярно перемножаемые на каждый запрос → сотни мс–секунды, особенно при конкурентной реиндексации. Это бьёт по целям «семантический поиск < 300ms» и «RAG до первого токена < 1s». Раздел 13 сам закладывает риск «sqlite-vec недостаточно быстр», но раздел 10 ему противоречит.
**Как исправить:** Убрать формулировку «ANN/HNSW». Раз цель «правильно сразу» — заложить настоящий ANN (usearch / hnsw_rs / Qdrant-embedded) уже сейчас, а не «fallback на фазе 1». Если остаётесь на sqlite-vec — обязательна метадата-префильтрация (папка/тег/дата) до KNN и честный пересчёт бюджетов раздела 10.

### Б7 · Центральный поток индексации невыразим: sqlx async-замыкание-транзакция не компилируется
**Нашли:** R (1, но фундаментально)
**Проблема:** `self.db.transaction(|tx| async move {...})` (4.2) — у sqlx нет такого API с async-замыканием в стабильном Rust (HRTB-лайфтайм `'c` не выводится через замыкание — известная многолетняя боль). Внутри `tx.upsert_file(...)` подаётся так, будто `tx` — ваш `DbQueries`, но sqlx-транзакция таких методов не имеет, а эксклюзивный `&mut tx` запрещает `tokio::join!` внутри. Как написано — не соберётся.
**Как исправить:** Перейти на `rusqlite` + выделенный write-actor (mpsc-канал → один поток с `Connection`); транзакция — синхронный `conn.transaction(|c| {...})`, тривиально атомарна и быстрее на мелких операциях. Async поверх single-writer SQLite даёт только оверхед. (См. также спор С1.)

### Б8 · Схема БД содержит нерабочие места: FTS5 на несуществующей колонке + осиротевшие векторы
**Нашли:** R, ML (2)
**Проблема:** (а) `fts_files USING fts5(title, content, content=files, content_rowid=id)` (раздел 5) — external-content FTS5 требует колонку `content` в таблице `files`, **которой в DDL `files` нет** (тело хранится на диске/в `chunks`). В таком виде FTS не заработает. Плюс не показаны обязательные триггеры синхронизации `AFTER INSERT/UPDATE/DELETE`. (б) `vec_chunks` — внешняя virtual table (vec0); SQLite `ON DELETE CASCADE` на неё **не распространяется**. Удалили файл → `chunks` уйдут каскадом, а строки в `vec_chunks` останутся «призраками» в выдаче. То же при reindex — старые векторы копятся.
**Как исправить:** FTS строить поверх `chunks.content` (`content=chunks`) или хранить тело в `files.content`; добавить триггеры синхронизации FTS. Добавить триггер `AFTER DELETE ON chunks → DELETE FROM vec_chunks WHERE chunk_id=old.id` и явную очистку векторов перед реиндексацией. (SA Agent синхронизирует FTS5 ровно триггерами.)

### Б9 · Идентичность файла рушится при атомарном сохранении + self-write watcher-цикл
**Нашли:** R, P (2)
**Проблема:** `VaultEvent::Modified → reindex_file` (4.2). Но почти все редакторы (и сам ваш фронт) сохраняют атомарно: пишут temp + `rename()` поверх. notify отдаёт это как `Create+Rename` или `Remove+Create`, **никогда** как чистый `Modified`. Файл будет удалён+создан → новый `file_id` → каскадное удаление `links/chunks/vec_chunks/file_tags`, на которых держится граф (petgraph `id_to_path`) и все FK. Граф рассыпается при каждом сохранении. Параллельно: `nexus.db` лежит **внутри** watched-vault, WAL постоянно пишет `*.db-wal/-shm/-vec` → рекурсивный watcher ловит собственные записи → цикл реиндексации (`.gitignore` тут не помогает, notify его не читает). Ещё и противоречие debounce: 300ms (4.2) против 500ms (6.1), и неверное «debounce встроен в notify» (нужен `notify-debouncer-full`).
**Как исправить:** Индексировать по нормализованному пути, не по событию: на `Create/Modify/Rename` делать reconcile «путь → запись в БД» и upsert, сохраняя `file_id`. Явный ignore-список в обработчике (`.nexus/`, `.git/`, `*.db*`, dot-файлы). Подключить `notify-debouncer-full`, зафиксировать одно окно. Тесты atomic-save на 3 ОС.

### Б10 · Стриминг чата сломан на уровне дизайна
**Нашли:** F, R (2)
**Проблема:** `useChat` (4.1): глобальный `listen('llm-stream')` без фильтра по сессии; `sessionId` генерится, но в payload чанка его нет и в обработчике не используется. Два чата (или чат + автосаммари + suggest, тоже через LLM) пишут в один `setResponse` → токены интерливятся. Ветка `done` только `return` — сообщение **никогда не финализируется** в историю (`chatMessages` пуст, при следующем `send` ответ исчезает). `setResponse(prev=>prev+chunk)` на каждый токен = O(n²) ре-рендеров + повторный парсинг всего markdown. `useEffect([])` + React 19 StrictMode → двойная подписка. Нет `isStreaming`/error/cancel. У вас уже есть правильный референс — SA Agent UI скоупит SSE по `jobId` с типизированными событиями `token/usage/done/error`.
**Как исправить:** Tauri 2 `Channel<StreamEvent>` на каждый `start_chat_stream` (привязан к вызову, решает корреляцию и порядок) либо `sessionId`+`seq` в payload. Аккумулятор в `useRef`, в state — throttled-снапшоты (rAF/≤100мс). Автомат `idle→streaming→done|error|cancelled`; на `done` атомарно коммитить в `chatMessages` + `chat_messages`. Команда `cancel_chat(sessionId)` + `CancellationToken` + `tokio::select!`.

### Б11 · WASM-«гарантии» (256MB, 5s CPU) не обеспечиваются кодом + path traversal в host-функциях
**Нашли:** R, S (2)
**Проблема:** 7.4 ставит `tokio::time::timeout(30s, instance.call(...))` — это (а) 30s, а не заявленные в 11 5s (прямое противоречие), и (б) `tokio::timeout` **не прерывает** синхронный WASM-цикл (`loop{}` не отдаёт управление) → DoS тривиален; нужен `epoch_interruption`/fuel. Лимит памяти 256MB не enforce'ится (нет `StoreLimits`/`ResourceLimiter`) → Vec-бомба роняет процесс с доступом ко всему vault. Главная же поверхность — host-функции: `vault.readFile(path)` без канонизации пути допускает `"../../.ssh/id_rsa"` и симлинки наружу; то же `write_file` = перезапись произвольных файлов ОС.
**Как исправить:** `Config::epoch_interruption` + `set_epoch_deadline` (или fuel) под реальные 5s; `StoreLimits` (memory + table/instance caps); один `Store`/краш-домен на плагин. Единая `resolve_vault_path()` для всех host-функций и Tauri-команд: `join → canonicalize → starts_with(vault_root)` + политика симлинков; покрыть тестом на `..`/абсолютный путь/симлинк.

### Б12 · Нет модели вкладок/сплитов — единственный `currentFile`
**Нашли:** F (1, но ломает ядро)
**Проблема:** `VaultStore` (4.1) хранит один `currentFile: NoteFile | null`. Ни вкладок, ни сплитов, ни нескольких открытых документов — притом что это базовая функция Obsidian, чей vault вы обещаете открывать. На одиночном `currentFile` завязаны контекст чата, suggest, backlink-панель. Добавление вкладок позже = переписывание store, модели фокуса, привязки AI-контекста и самого редактора (несколько инстансов CM6). Это ретрофит уровня ядра — ровно то, что принцип «правильно сразу» велит избегать.
**Как исправить:** Уже в Фазе 0 ввести `editorGroups:{id, tabs:TabId[], activeTab}[]`, `activeGroupId`, документы как открытые буферы; AI-контекст/backlinks брать из активного таба.

## Серьёзные проблемы (исправить в фазе 0–1)

### С-1 · `git2-rs` блокирует async-рантайм
**Нашли:** R. `auto_commit` (8.2) помечена `async`, но внутри блокирующий libgit2 (`statuses`, `add_all`, `commit`, сетевые `pull/push`). `add_all` на 50k файлов = секунды синхронной работы → заморозка всех IPC. → Все git2-вызовы в `spawn_blocking` или выделенный git-поток; CPU-bound parse/blake3/chunk — тоже через `spawn_blocking`/rayon.

### С-2 · Fallback провайдера ловит только ошибку создания стрима + тихо сливает приватные данные
**Нашли:** R, P, S. `AIClient::complete` (4.3) матчит `Err` только на этапе коннекта; реальные обрывы инференса приходят **внутри** `Stream::Item` и фоллбэк их не видит — а это самый частый кейс локальной модели. Хуже: авто-переключение на OpenAI/Anthropic при сбое llama.cpp **молча** отправляет контент vault в облако, нарушая «Privacy by default». → Оборачивать стрим, фоллбэк только до первого выданного токена; на облако — только при явном opt-in именно на fallback, с видимой индикацией «отвечает облако» и пересчётом контекст-бюджета под чужое окно.

### С-3 · Cross-encoder reranker «через llama.cpp <200ms» нереализуем
**Нашли:** ML, R. На :8080 крутится генеративная Qwen3-27B; reranking llama.cpp даёт только с отдельной reranker-моделью (`--reranking`), а через 27B «<200ms» физически невозможно (десятки секунд). → Либо отдельный инстанс bge-reranker на своём порту (как :8081/:8082 в SA Agent) c SLA ~1s, либо LLM-listwise rerank как в SA Agent; убрать «<200ms».

### С-4 · Graph Boost `+0.2` поверх RRF-скоров — арифметически переопределяет ранжирование
**Нашли:** ML. RRF-скор топ-чанка ≈ `1/(60+1)≈0.016`; прибавка +0.2 в 6–12 раз больше всего скора → любой чанк из соседнего по графу файла вылетает на первое место независимо от релевантности. Тихо ломает качество в самом частом сценарии (открыт файл со ссылками). → Граф как третий ранг в самой RRF-формуле или мультипликативный множитель в шкале RRF; зафиксировать тестом на eval.

### С-5 · Чанкер: токены без токенайзера, overlap сверх лимита, разметка загрязняет вектор
**Нашли:** ML. (а) «512 токенов» считаются без токенайзера — для кириллицы ошибка 1.5–2× → переполнение/недозаполнение бюджета 6.4. (б) `add_overlap` добавляет 64 токена **после** нарезки → фактический чанк 576, `token_count` врёт. (в) Нет вырезания frontmatter, защиты code-block/таблиц от разрыва. → Токенайзер **эмбеддера** (через `/tokenize`); overlap закладывать внутрь окна; frontmatter вон из тела, fenced-code/таблицы атомарны.

### С-6 · Асимметрия query/document эмбеддинга и нормализация проигнорированы
**Нашли:** ML. `nomic-embed-text` требует префиксы `search_query:`/`search_document:`; без них качество падает. Косинус в sqlite-vec предполагает L2-нормализацию, а llama.cpp её не гарантирует. SA Agent явно нормализует query. → `embed_query`/`embed_document` с правильными префиксами; нормализовать в Rust явно.

### С-7 · Дедуп чанков по overlap + «средний вектор файла» для suggest
**Нашли:** ML. Топ-8 чанков (6.4) легко включат два соседних с overlap → дубль в контексте, бюджет посчитан как уникальный. «Среднее по чанкам» (6.3) для многотемной заметки даёт центроид «средней температуры», порог 0.75 не работает; колонки файлового вектора в схеме вообще нет. → Мерж смежных чанков по `char_start/char_end`; suggest — max-sim по чанкам, а не файл↔файл.

### С-8 · `suggest_on_save:true` → LLM-reasoning на каждое сохранение
**Нашли:** ML. Связка с 6.3-режим-2 (~2–5s на 27B) забьёт единственную очередь инференса, затормозит чат, нашумит подсказками на недописанной заметке, а с облаком — реальные деньги на каждый ⌘S. → `suggest_on_save:false` по умолчанию; LLM-режим по явному действию или idle-debounce 30–60с с дедупом по хэшу; разнести очереди embed/chat/suggest.

### С-9 · Дерево 50k: react-virtual решает не ту задачу + мегабайтный IPC
**Нашли:** F, R. `@tanstack/react-virtual` виртуализирует плоский список, а нужно дерево с expand/collapse/фильтром/DnD (нужен flatten-слой видимых узлов). `listFiles(vault)→FileEntry[]` тянет 50k объектов одним IPC = десятки МБ JSON, бьёт по «старт < 3s». То же `get_graph()`. → Ленивый `listDir(path)` + flatten поверх react-virtual; поиск/фильтр на стороне Rust по таблице `files`; пагинация/бинарный канал для крупных payload.

### С-10 · Граф: layout ≠ render, блокировка main-thread, два движка
**Нашли:** F. Узкое место — не отрисовка, а force-layout на 50k нод (секунды-минуты, блокирует main-thread); «рендер < 2s» меряет не то. D3 (<5k) + sigma.js (>5k) = две реализации и два overlay-API для плагинов. → Один движок (sigma/WebGL), layout в Web Worker (graphology) или предрасчёт в Rust; основной режим для больших vault — локальный N-hop граф.

### С-11 · i18n: русские плюралы сломаны, нет Intl-форматирования и Collator
**Нашли:** F. `"Indexing {{count}} files"` без плюральных категорий → «проиндексировано 5 файл». Нет локале-форматирования чисел/дат и `Intl.Collator('ru')` для сортировки 50k файлов (кириллица). → Плюральные ключи `_one/_few/_many`, `Intl.NumberFormat/DateTimeFormat`, `Intl.Collator(locale,{numeric:true})`, детекция системной локали при первом запуске.

### С-12 · i18n Rust-стороны и плагинов не спроектирован; кросс-язычный RAG не работает
**Нашли:** F, P, ML. «Rust тоже локализует» — одна фраза без механизма (ошибки индексатора/git/AI будут на английском). SDK не имеет i18n-API для строк плагинов. И отдельно: `nomic-embed-text` слабо мультиязычен → запрос на русском по англоязычной заметке даст низкий similarity. → Бэкенд-i18n (fluent/rust-i18n) с общим источником ключей; i18n-API в SDK с namespacing `plugin:<id>:<key>`; **мультиязычный эмбеддер** (bge-m3 / multilingual-e5) под требование RU/EN.

### С-13 · Versioning Plugin API сломан семантически
**Нашли:** P, R. `api_version:"^1.0"` = «любой 1.x», но editor-extensions/`vault:watch` появляются в v1.2 (Appendix B) → плагин под v1.2 установится на ядро 1.0 и упадёт в рантайме. Нужен **минимум** версии, а не каретный диапазон. Нет capability-detection, deprecation-политики, версионирования самого CodeMirror 6 (дублирование `@codemirror/state` роняет редактор). Appendix B сам себе противоречит (editor-extensions то в v1.0/Фаза 2, то в v1.2). → `min_api_version`(+`max`), `api.capabilities`, deprecation-policy, реэкспорт CM6 из SDK с жёстким пином.

### С-14 · First-party «плагины» упрутся в собственную песочницу
**Нашли:** P, F. git-sync требует сети/FS/credentials вне vault — WASM-песочница (нет FS/сети, 256MB, 5s) этого не даёт → это native-код, замаскированный под плагин. Граф 50k на WebGL несовместим с CPU-лимитом 5s/вызов и opaque-origin iframe. → Честно ввести категорию «core modules» (привилегированные, не плагины) ИЛИ доказать, что публичная модель достаточно мощна. Сейчас терминология маскирует, что модель неполна.

### С-15 · Холодный старт < 3s противоречит модели; реиндекс 50k < 30 мин нереалистичен
**Нашли:** R, ML, F. В бюджет 3s заложены hydrate графа (заявленные «50ms» для 500k рёбер — фантазия, реально сотни мс–секунды), `load_extension(sqlite-vec)`, cold-compile WASM-модулей (Cranelift, сотни мс/модуль). «50k файлов < 30 мин»: 50k → 150–400k чанков, при ~100–200 эмбеддингов/с это 25 мин–часы только на инференс. → Разнести бюджет «UI интерактивен» (мгновенно) и «индекс/граф готов» (фон); precompile WASM в `.cwasm`; бюджет считать в чанках с измеренным throughput; прогресс «N/M чанков», а не «<30 мин».

### С-16 · `auto_commit` через `add_all(["*"])` коммитит всё, включая секреты
**Нашли:** S, R, ML. Коммитит любой положенный в vault `.env`/токен/скриншот в git-историю навсегда и на все устройства; сообщение коммита (из `statuses`) может не соответствовать реально закоммиченному; затрёт ручной стейджинг; авто-коммит каждые 300с = свалка истории. → Выборочный stage только трекаемых `.md`/одобренных типов; secret-scan + denylist перед коммитом; коммит по idle + squash.

### С-17 · `auto_pull` + watcher против открытого редактора → потеря правок и порча индекса
**Нашли:** R, F. Фоновый pull переписывает `.md` на диске → watcher → массовая реиндексация → коммит полусмерженного. Нет протокола «диск изменился под грязным буфером CodeMirror» → прямой путь к потере несохранённого. `.conflict`-файлы (8.3) не в ignore — попадут в FTS/граф мусором. → Sync-lock (на время pull приостановить watcher/индексацию); если буфер чист — перечитать с диска, если грязен — diff/prompt; `.conflict` в ignore.

### С-18 · SSRF через `llamacpp.base_url` + отсутствие auth на локальном llama.cpp
**Нашли:** S. `config.json` синхронизируется через git; `base_url` без валидации → подменённый `http://169.254.169.254/...` или внутренний сервис = SSRF руками хоста. Реальные серверы (из SA Agent) без auth на LAN → любой плагин/конфиг свободно дёргает чужой инференс. → Allowlist схем/хостов (по умолчанию loopback), запрет приватных/metadata-диапазонов без согласия; изменённый `base_url` из pull не применять без подтверждения; auth-токен к llama.cpp в keychain.

### С-19 · Prompt injection через индексированные заметки
**Нашли:** S. RAG (6.2) и suggest-LLM (6.3) кладут контент заметок в промпт; обёртка `<note>…</note>` ломается контентом с `</note>`. Синхронизированный/импортированный/скачанный vault с «Ignore previous instructions…» отравляет подсказки всем; ломает JSON в `complete_json`; на будущее (tool-use, v2.0) — опаснее. → Неподделываемые разделители + инструкция «между маркерами — данные»; строгая валидация JSON-ответа; пометка контента из непроверенных источников. Заложить **до** tool-use.

### С-20 · Tauri IPC из iframe-плагинов, CSP не специфицированы
**Нашли:** S. Если `invoke` доступен глобально в webview, плагинный iframe зовёт `read_file`/`write_file`/`git_*` напрямую в обход всего postMessage-моста и permissions. CSP, capabilities-allowlist, `dangerousRemoteDomainIpcAccess` не описаны. → Плагинные iframe строго вне зоны прямого `invoke`; минимальный capabilities-allowlist (ни одной vault-команды webview-контексту плагина); жёсткий CSP без inline/eval. Аудит `tauri.conf.json` — в Фазу 0.

### С-21 · CM6 ↔ React: владение DOM и петли не зафиксированы; сохранение/«грязный» буфер не описаны
**Нашли:** F. Не определено, кто владеет DOM редактора; риск цикла CM→onChange→setState→re-render→новый doc→… Нет автосохранения, понятия dirty, разрешения гонки watcher↔буфер. → `EditorView` создаётся один раз в `useLayoutEffect`, смена файла через `dispatch(changes)`, а не пересоздание; однонаправленный поток с подавлением эха; guard против StrictMode double-mount.

### С-22 · Live Preview уровня Obsidian недооценён на порядок
**Нашли:** F. В таблице рисков его нет вовсе. → Вынести в отдельный эпик; в Фазе 0 — source-mode + подсветка + клик по wikilink; добавить строку в раздел 13 (Вероятность Высокая / Влияние Высокое).

## Решения под вопросом
Места, где выбор сделан, но есть сильная альтернатива, либо где ревьюеры **разошлись**.

### СП-1 · WASM-изоляция против JS-DX — прямой конфликт «безопасность ↔ экосистема» (агенты разошлись)
Единственное место, где роли тянут в разные стороны. **S** ценит WASM как единственный слой, реально песочащий CPU/память untrusted-кода. **R/P/F** в один голос: WASM-first убивает порог входа и cold-start экосистемы (Obsidian взлетел именно на «обычный JS + прямой доступ к app»), а раздел 13 сам метит риск «экосистема не появится» как Высокий — то есть архитектура заложила свой главный риск в фундамент. *Разрешение:* JS/TS как основной путь с **строгим host-side broker'ом** (а не sandbox-iframe как мнимой границей), процессная/worker-изоляция, опциональный WASM для тяжёлой/доверенной логики. Это удовлетворяет и DX (P/F/R), и значительную часть требований S — при условии, что broker, аудит и path-scoped permissions реализованы (Б2).

### СП-2 · sqlx против rusqlite
**За sqlx:** compile-time `query!`, единый async-стиль. **Против (R):** SQLite single-writer — async-слой это чистый оверхед, пул к одному файлу даёт `SQLITE_BUSY`, async-замыкание-транзакция не выражается (Б7). Рекомендация R — rusqlite + write-actor. Решение влияет на Б4/Б7/С-1, принять его лучше до кода.

### СП-3 · petgraph in-memory как источник истины для беклинков
**За:** O(1) беклинки. **Против (R, частично ML/F):** дублирование с таблицей `links` → рассинхрон (три отдельных лока `RwLock`+2×`DashMap`), фантомные «50ms/100MB», холодный hydrate в бюджете старта. Беклинки `WHERE target_id=?` по существующему индексу `idx_links_target` — это доли мс из page-cache без рисков. *Разрешение:* понизить petgraph до опционального кэша для тяжёлых графовых алгоритмов (которых в требованиях нет), источник истины — SQLite.

### СП-4 · Единый `nexus.db` для индекса + векторов + истории чатов
Сошлись три угла с разной мотивацией: **ML** — тяжёлый embed/reindex лочит читающие запросы UI; **F** — история чатов лежит в .gitignore'нутом, пересоздаваемом индексе → теряется при ребилде и не синхронизируется между устройствами; **S** — единый writer = единая точка DoS, чувствительная история чата рядом с индексом. *Разрешение:* вынести историю чатов в отдельное (синхронизируемое и/или шифруемое) хранилище — либо отдельная БД, либо `.md` в vault (в духе plain-files); векторы — в отдельный файл/индекс, чтобы не лочить основную БД.

### СП-5 · Tauri events против `Channel` для стриминга
**За events:** просто, broadcast. **Против (R, F):** нет привязки к вызову, нет порядка между сессиями, JSON на каждый токен через общий bus (корень Б10). Для пер-сессийного стрима — `Channel<T>`; events оставить для одной глобальной нотификации (прогресс индексации).

### СП-6 · Один `LLMProvider` для chat и embeddings; один хост llama.cpp
**Против (ML):** Anthropic не имеет embeddings — что вернёт `AnthropicProvider::embed`? Chat и embeddings — разные модели/хосты/жизненный цикл; SA Agent физически разносит (чат удалённо `192.168.0.172`, эмбеддинги локально `:8081/:8082`), потому что одна модель-слот не обслуживает эффективно и генерацию 27B, и батч-эмбеддинг. *Разрешение:* два трейта `ChatProvider`/`EmbeddingProvider`, раздельные `chat_url`/`embedding_url` в конфиге + health-check эмбеддера (он, по опыту SA Agent, «слетает после ребута»).

### СП-7 · `config.json` с security-полями в git под политикой «Ours»
**За:** локальная конфигурация приоритетнее. **Против (S):** `base_url`, `fallback_provider`, `plugins.enabled`, `auto_pull` — поля доверия; синхронизировать их по тем же правилам, что заметки, рискованно. *Разрешение:* security-поля — локально (keychain/локальный settings), в git — только несекьюрные предпочтения.

## Пробелы в плане
Отсутствует, при этом всплыло у нескольких ревьюеров или критично у одного.

- **Eval/quality-харнесс для RAG** (ML). Нет recall@k/nDCG/MRR, golden-набора, регрессии — а решения «RRF top-20 / порог 0.75 / +0.2 / 8 чанков» приняты без способа проверить. Без этого «Correctness over speed» неисполним. У вас **уже есть** канон — `sa-eval` (см. ниже).
- **Миграции схемы и переэмбеддизация при смене модели** (R, ML, F). Нет раннера; vec0/FTS5 нельзя `ALTER`; смена `embedding_model` тихо инвалидирует все векторы.
- **Восстановление после краха посреди индексации** (R). Нет reconcile-прохода по `indexed_at`<`updated_at`, нет «грязного» флага, нет переживающей рестарт очереди.
- **Не-markdown и вложения** (ML, F). PDF/изображения/аудио, embeds `![[file]]`, Mermaid, LaTeX, файлы 1–5 МБ (не влезут в эмбеддер). В плане только `.md`.
- **Жизненный цикл и дистрибуция плагинов** (P, S). update/rollback/uninstall/`onUninstall`-cleanup, межплагинные зависимости, миграция settings, где хостятся плагины, привязка `id`→издатель, целостность/хэши, blocklist/kill-switch, изоляция данных между плагинами.
- **Command registry и клавиатурная модель как ядро** (F). `registerCommand` есть, но реестр (на котором держатся и ядро, и плагины) и keymap отложены в Фазу 4 — плагинный API Фазы 2 спроектируется вслепую. Туда же — a11y (роли дерева, фокус, скринридеры).
- **Security-кластер** (S): отдельного раздела «Модель угроз» нет; at-rest шифрование (`nexus.db` хранит полный контент в FTS/чанках + историю чата открытым текстом); enforcement «логи без контента» (пути — уже PII) и scrubbing crash-reporter'а; «секретные» папки, исключаемые из индекса/RAG/плагинов; реальное удаление (soft-delete оставляет контент в FTS/векторах/git-истории навсегда); целостность git-remote (host-key pinning, подпись коммитов); deep-link/file-association (открыть чужой vault = запустить его `.nexus/plugins`).
- **Нормализация путей и регистр ФС** (R). macOS case-insensitive vs Linux; `path TEXT UNIQUE` создаст дубли; wikilink-резолвинг (Obsidian-совместимость) не описан.
- **Производительность при множестве плагинов и тема в iframe** (P, F). Метрики раздела 10 для «голого» ядра; 20 плагинов с подписками на каждое изменение файла не оценены. CSS-переменные темы не протекают в opaque-origin iframe → плагинный UI не подхватит тёмный режим; FOUC при старте.
- **Виртуализация ленты чата и панели результатов/беклинков** (F). Тысячи результатов/длинная история не виртуализированы; backpressure стрима (27B быстрее, чем рендерит React).

## Вердикт

**Нужен revision pass перед стартом кодинга — не косметический, а по 3–4 архитектурным развилкам.** Скелет хорош: local-first, plain-files, слоистость, provider-абстракция, фазовость — всё это здравое и достойно реализации. Но в текущем виде план не проходит собственный принцип «правильная архитектура сразу» по двум фронтам. Во-первых, **ядро индексации и RAG содержит ошибки уровня «не соберётся / не заработает»**: эмбеддинг файла вместо чанков, несовпадение размерности 1024/768, FTS5 на несуществующей колонке, осиротевшие векторы, sqlx-транзакция-замыкание, sqlite-vec, выдаваемый за ANN, поломка идентичности файла при атомарном сохранении. Во-вторых, и это важнее, **плагинная подсистема — ключевое non-negotiable требование — внутренне противоречива** (WASM-рантайм против JS-SDK с CodeMirror-расширениями), **небезопасна by design** (sandbox как мнимая граница, неаутентифицируемый postMessage, git-доставка неподписанного исполняемого кода, неработающие WASM-лимиты, path traversal) и **не запускает экосистему** (WASM-барьер входа — риск, который план сам метит Высоким). Эти решения дешевле всего изменить сейчас, на бумаге, а не после Фазы 2. Рекомендую сфокусировать revision на: (1) единая плагинная модель — JS-first с реальным host-broker'ом, аудитом и path-scoped permissions; (2) корректность RAG — чанк-уровневые эмбеддинги, динамическая размерность, hybrid/rerank по образцу SA Agent, eval-харнесс; (3) исправление схемы БД и выбор rusqlite + write-actor; (4) раздел «Модель угроз» и перенос подписи плагинов в Фазу 2. Остальное (стриминг, дерево, граф, i18n, вкладки) — серьёзно, но решаемо в Фазе 0–1 без переписывания фундамента.

## Приложение · Переиспользование SA Agent (ответ на опциональный вопрос)
Существующий проект `SA Agent` — это не помеха, а готовый набор уже оплаченных болью уроков, прямо применимых к слою RAG Nexus (углубился ML-ревьюер):
- **`sa-eval`** — готовый eval-харнесс (`inputs/`, `tests/`, `comparisons/`, правило «сравнение валидно только при совпадении условий», `attribution_eval.py`). Закрывает главный пробел Nexus — переносить методологию, а не изобретать.
- **Чанк-уровневые эмбеддинги** и удаление старых векторов перед реиндексацией — ровно фикс Б4.
- **Урок размерности** (BGE 1024 → nomic/jina 768, авто-скип несовпадающих) — это Б5, уже прожито.
- **Раздельные хосты** (чат удалённо, эмбеддинги локально `:8081/:8082`) и знание, что embed-серверы «слетают после ребута» → health-check — это СП-6 и С-18.
- **LLM-as-reranker** вместо несуществующего cross-encoder-эндпоинта — это С-3.
- **Явная L2-нормализация query** перед косинусом — это С-6.
- **FTS5 синхронизация триггерами** `AFTER INSERT/UPDATE/DELETE` — это Б8.
- **Типизированный SSE по `jobId`** (`token/usage/done/error`) из `SA Agent - UI` — это правильный референс для Б10.

Конкретный нюанс: SA Agent индексирует код (чанкинг по `func/type`/`def/class`), Nexus — markdown, поэтому переиспользуется **методология и инфраструктурные клиенты** (eval, llama-клиенты с таймаутами/retry, нормализация, FTS-триггеры), а не доменный чанкер один-в-один.
