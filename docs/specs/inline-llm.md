# Vision→AC: Inline LLM в редакторе

> **Что это.** Перевод vision-фичи «Inline LLM» (BACKLOG §Идеи) в **реализуемую спеку**: измеримые AC,
> определение «что тестируем / что нет», и зафиксированные продуктовые решения. Это снимает блокер из
> ревью §2/A2 («у фич нет AC + зашиты продуктовые решения владельца» → сперва vision→AC, потом код).
>
> **UX/визуал — НЕ здесь:** полностью описан в `docs/design/DESIGN_BRIEF.md` §4 (ghost-text, inline-тулбар
> по выделению, панель структуры, slash-меню, состояния, токены, a11y). Эта спека — **контракт поведения**
> и **тест-стратегия** поверх готового UX.
>
> **Статус:** ✅ продуктовые решения D1–D5 зафиксированы владельцем (см. ниже). Готово к нарезке на срезы.

## 1. Скоуп v1 и non-goals

**v1 (в этой спеке):**
- Ручные триггеры: **slash-команда** `/`, **inline-тулбар по выделению**, явная команда «продолжить».
- **Ghost-text** как способ показа предложения: серый текст, `Tab` принять / `Esc` отклонить.
- Стриминг результата в ghost (переиспользуем chat-стрим + rAF-троттл, как чат V2.4).
- Контекст генерации — **текущая заметка** (текст вокруг курсора / выделение), без RAG.

**Non-goals v1 (отдельные спеки / срезы, «no silent caps»):**
- **RAG-грунтинг inline-генерации** (контекст из всего vault) — отдельно, см. D2.
- **Авто-ghost по паузе в печати** — см. D1 (по умолчанию выкл).
- **Обучение на принятых/отклонённых** (логирование письма) — см. D3 (отложено).
- **Панель выбора структуры / `/template`** — пересекается с фичей «Умные шаблоны» → её отдельная
  vision→AC спека (классификатор типа заметки). Здесь только заглушка-триггер.
- **Web/tool-use** — BLOCKED (egress-ADR + SearXNG).

## 2. Продуктовые решения (зафиксированы владельцем)

> Это «зашитые продуктовые решения», которые ревью требовало вынести явно. Значения по умолчанию —
> в настройках (раздел настроек, BACKLOG), но дефолт фиксируется здесь.

| ID | Решение | Значение (зафиксировано) |
|----|---------|--------------------------|
| **D1** | Авто-ghost по паузе письма (~2с) | **ВЫКЛ по умолчанию** (opt-in в настройках) ✅ |
| **D2** | Контекст генерации | **Текущая заметка** (вокруг курсора / выделение) ✅ |
| **D3** | Логировать принятые/отклонённые «для обучения» | **Отложить** (не персистим) ✅ |
| **D4** | Набор inline-действий по выделению (v1) | Continue · Rewrite · Summarize (рек.) |
| **D5** | Набор slash-команд (v1) | `/continue` · `/rewrite` · `/summarize` · `/ask` · `/template`(→ умные шаблоны) (рек.) |

**Обоснование зафиксированного:**
- **D1 — ВЫКЛ по умолчанию.** Ручные триггеры (slash/выделение) всегда вкл; авто-ghost по паузе — opt-in.
  Авто-предложения при *письме* (не коде) сбивают поток мысли и жгут локальную модель на каждой паузе.
  Если пользователь включит — дебаунс 2с (DESIGN_BRIEF §4.5). Пересмотр — после реального использования.
- **D2 — текущая заметка** (±контекст вокруг курсора / выделение). `continue`/`rewrite`/`summarize`
  локальны по смыслу; RAG-грунтинг (контекст из всего vault) = +стоимость/латентность/сложность → отдельный
  срез, когда будет спрос на «продолжи, опираясь на мои заметки про X».
- **D3 — отложить логирование.** Нет консьюмера-обучения; хранить черновики письма = новая
  приватность-поверхность. Когда появится механизм обучения — отдельное решение + обязательно `Redacted`
  (AC-SEC-6) для контента. Сейчас принятие/отклонение нигде не персистится.

## 3. Acceptance Criteria (Given/When/Then)

> **Тестируем СТРУКТУРУ/механику** (детерминированно, через мок chat-провайдера), НЕ качество вывода LLM.
> Нумерация `AC-IL-*`; переедут в `docs/acceptance/ACCEPTANCE.md` при старте реализации.

- **AC-IL-1 (триггер→запрос)** *Given* открыт документ *When* пользователь вызывает inline-генерацию
  (slash-команда / действие тулбара) *Then* в chat-провайдер уходит ОДИН запрос с собранным контекстом
  (текущая заметка по D2), статус «генерируется» виден у курсора.
- **AC-IL-2 (стрим→ghost)** *When* приходят токены ответа *Then* они показываются как ghost-text
  (стилем faint), применяются **батчем по кадру** (rAF-троттл, ≤~60 ре-рендеров/с — как чат AC-Б10-4),
  не по символу.
- **AC-IL-3 (accept)** *Given* активен ghost-text *When* `Tab` *Then* ghost фиксируется как обычный
  текст документа в позиции вставки, документ помечается dirty, ghost-состояние очищается.
- **AC-IL-4 (reject)** *Given* активен ghost-text *When* `Esc` *Then* ghost исчезает, документ и позиция
  курсора не изменены (как до предложения).
- **AC-IL-5 (Tab-роутинг)** *Given* ghost НЕ активен *When* `Tab` *Then* обычная табуляция (inline-фича
  не перехватывает Tab без активного предложения).
- **AC-IL-6 (отмена стрима)** *Given* идёт стрим *When* `Esc` *Then* стрим прерывается (как
  `chat_cancel`), ghost очищается, накопленный текст НЕ вставляется.
- **AC-IL-7 (модель недоступна)** *Given* chat-провайдер не сконфигурирован/недоступен *When* триггер
  *Then* тихая inline-нотификация-ошибка у курсора, БЕЗ модала и БЕЗ краша; редактор полностью рабочий.
- **AC-IL-8 (один активный стрим)** *Given* идёт inline-стрим *When* новый триггер *Then* предыдущий
  отменяется (один активный inline-запрос за раз, как чат `AppState::begin_chat`).
- **AC-IL-9 (выделение→тулбар→результат)** *Given* есть выделение *When* пользователь жмёт действие
  тулбара (Rewrite/Summarize/Continue) *Then* результат показывается ghost-ом с accept/reject; accept
  заменяет/дополняет выделение (Rewrite/Summarize — заменяет; Continue — вставляет после).
- **AC-IL-10 (a11y)** ghost анонсируется `aria-live="polite"` один раз (не по символам); хинт
  «Tab принять/Esc отклонить» виден текстом; принятие/отклонение доступно с клавиатуры (основной путь).

## 4. Измеримый успех / что НЕ тестируем

- **Тестируем (зелёный CI, детерминированно):** все AC-IL-1..10 — через **мок chat-провайдера**
  (поток sources?/token/done/error управляем). Это механика: триггер→запрос, стрим→ghost, accept/reject,
  Tab-роутинг, отмена, обработка ошибки, один-стрим, a11y-атрибуты. Frontend (CM6 + стор) + Rust (сборка
  контекста, команда).
- **НЕ тестируем зелёным CI (вероятностный вывод LLM):** качество/релевантность/связность предложений —
  это **human-eval**, не автотест. Прямо фиксируем, чтобы не было «зелёных тестов на бессмысленный вывод»
  (ревью). Опционально — отдельный ручной чек-лист качества при ручном тестировании фазы.

## 5. Зависимости / архитектура

- **CM6-расширение:** ghost-text как decoration (atomic widget / mark), keymap с приоритетом `Tab`/`Esc`
  над дефолтными ТОЛЬКО при активном ghost (AC-IL-5).
- **Chat-провайдер (ADR-005):** переиспользуем — отдельная команда `inline_complete`(channel, prompt,
  selection?, mode) поверх `ChatProvider::stream_chat`; тот же rAF-троттл, что чат (V2.4); отмена как
  `chat_cancel`.
- **Промпт:** системная инструкция по режиму (continue/rewrite/summarize) + контент по D2. Анти-инъекция
  (AC-SEC-7): контент заметки — данные; при D2=текущая-заметка инъекция-риск ниже (свой документ), но
  системную рамку «между маркерами — данные» переиспользовать из `build_*_messages`.
- **Egress:** НЕ нужен (локальная модель). RAG-грунтинг (D2-alt) / web — отдельный egress-ADR.
- **Стор inline-сессии** (один активный стрим; буфер rAF), по образцу `stores/chat.ts`.

## 6. Нарезка на срезы (после фиксации D1–D5)

1. **IL-1 Бэкенд:** команда `inline_complete` (стрим + отмена) + сборка контекста текущей заметки +
   режимы continue/rewrite/summarize + Rust-тесты (мок провайдер).
2. **IL-2 CM6 ghost-core:** decoration ghost-text + keymap Tab/Esc (роутинг) + стор + стрим/rAF +
   accept/reject/cancel + фронт-тесты (AC-IL-1..8).
3. **IL-3 Триггеры UX:** slash-меню (D5) + inline-тулбар по выделению (D4) + ошибка/a11y (AC-IL-9/10).
4. **IL-4 (опц., по D1):** авто-ghost по паузе + настройка вкл/выкл.

## 7. Открытые вопросы

- D1/D2/D3 — ждут подтверждения владельца (см. §2). D4/D5 — рекомендованы, легко скорректировать.
- «Умные шаблоны» (`/template` + панель структуры) — **отдельная vision→AC спека** (классификатор типа
  заметки: эвристика→LLM, пороги уверенности, обучение на принятых). Здесь только триггер-заглушка.
