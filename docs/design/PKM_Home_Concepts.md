# PKM Home Page — концепция и логика

Форк Obsidian с упором на локальные LLM. Home Page — персонализированный дашборд, который агрегирует состояние базы знаний и точечно усиливает её через LLM.

---

## Принципы

- **Инди-инструмент.** Один пользователь, никакой избыточности. Всё заточено под личный PKM / second brain.
- **LLM вспомогательный.** Модель подключается точечно — там где даёт реальный прирост. Не доминирует над интерфейсом.
- **Скорость важна.** Статические виджеты загружаются мгновенно. LLM-виджеты — асинхронно, с кэшем, не блокируют страницу.
- **Конфигурируемость.** Состав и расположение виджетов настраиваются пользователем.

---

## Три класса виджетов

**Статические** — данные берутся напрямую из vault без какой-либо обработки. Срабатывают мгновенно при любом изменении файлов.

**Динамические** — вычисляются из метаданных и frontmatter. Не требуют LLM, но применяют логику поверх сырых данных (scoring, агрегация, фильтрация).

**LLM-powered** — генерируются локальной моделью. Результат кэшируется и обновляется по триггеру или вручную. Никогда не блокируют загрузку страницы.

---

## Режимы обновления LLM-виджетов

- **on-open** — при каждом открытии дашборда, если кэш устарел
- **on-vault-change** — реагирует на изменение конкретных файлов (с задержкой ~30 секунд чтобы не срабатывать на каждый символ)
- **scheduled** — раз в сутки в фоне, пока приложение открыто
- **manual** — только по кнопке пользователя; для тяжёлых или нечастых операций

## Структура страницы

Страница делится на четыре смысловые зоны сверху вниз.

### Зона 1 — Ориентация

Всегда видима, не конфигурируется. Отвечает на вопрос «где я сейчас».

**Строка состояния** содержит: приветствие с именем, текущую дату, количество заметок в базе, статус подключения к локальной модели, путь к vault, количество изменений за сегодня и текущую серию дней.

**Строка поиска** — единая точка входа для двух режимов. Префикс `/` запускает поиск по vault, префикс `?` отправляет вопрос к LLM. Визуально один элемент, логически два разных действия.

**Быстрые действия** — набор кнопок для частых операций: создать заметку, открыть daily note, захватить мысль, открыть граф, обновить все LLM-виджеты.

---

### Зона 2 — Сводка дня

Отвечает на вопрос «что происходит прямо сейчас».

#### Daily brief
*Класс: LLM-powered. Триггер: on-open.*

LLM анализирует изменения vault за последние 24 часа и формирует краткую текстовую сводку: какие темы были активны, что появилось нового, на что стоит обратить внимание. Дополнительно извлекает активные теги периода.

Ключевая ценность — не просто список изменённых файлов (это даёт статистика), а осмысленная интерпретация: «ты работал над X, появилась заметка про Y, давно не открывал Z».

#### Недавние файлы
*Класс: статический. Триггер: vault.on(modify).*

Топ-7 файлов по времени последнего изменения. Для каждого: название, папка/тег, время изменения. Кликабельно — открывает файл.

---

### Зона 3 — Проекты и состояние базы

Отвечает на вопрос «куда я двигаюсь».

#### Прогресс целей
*Класс: динамический. Триггер: vault.on(modify).*

Прогресс-бары для заметок с тегом `#goal`. Значение прогресса берётся из поля `progress` в frontmatter (0–100). Никакого ручного ввода — файл изменился, frontmatter обновился, виджет пересчитался.

#### Статистика базы
*Класс: статический. Триггер: on-open.*

Четыре числа: всего заметок, изменений сегодня, заметок без обратных ссылок, текущая серия дней. Даёт быстрое ощущение «здоровья» базы без необходимости куда-то переходить.

---

### Зона 4 — Требует внимания

Отвечает на вопрос «что упускается».

#### Stale radar
*Класс: динамический + LLM-powered опционально. Триггер: on-open (слой 1), manual (слой 2).*

Двухслойная система обнаружения устаревших заметок.

**Слой 1 — scoring без LLM.** Каждая заметка получает числовой балл устаревания на основе нескольких сигналов:
- Количество дней без изменений (главный сигнал)
- Статус в frontmatter: черновик (`draft`) или в процессе (`wip`) добавляют баллы
- Прошедший дедлайн из поля `due` добавляет баллы
- Отсутствие обратных ссылок добавляет баллы
- Пометка `evergreen` снижает балл — вечнозелёные заметки не устаревают по определению
- Файлы в папках Templates/ и Archives/ исключаются полностью

Результат — ранжированный список с цветовой индикацией: красный (критично устарело), оранжевый (стоит проверить).

**Слой 2 — LLM-обогащение.** Только для заметок с высоким баллом (топ-10). LLM читает содержимое и даёт: одно предложение о причине устаревания, рекомендованное действие (обновить / архивировать / разбить / удалить), конкретную подсказку. Результат кэшируется на 24 часа и инвалидируется только если файл изменился.

#### Открытые вопросы
*Класс: LLM-powered. Триггер: manual.*

LLM сканирует последние 20 изменённых заметок и извлекает незакрытые вопросы — те, на которые нет ответа в тексте: риторические вопросы, незавершённые мысли, «надо разобраться». Выводится списком; клик на вопрос открывает исходную заметку.

---

### Зона 5 — Аналитика (опционально)

#### Context drift
*Класс: LLM-powered. Триггер: scheduled.*

LLM сравнивает две выборки заметок: последние N изменённых файлов (текущий фокус) и файлы с тегом `#goal` или `#priority` (долгосрочные цели). Находит расхождение и формулирует его в одном абзаце.

Ценность в том, что это мета-наблюдение: не «что ты делал», а «насколько то, что ты делал, соответствует тому, что ты хотел делать». Обновляется раз в день — чаще нет смысла.

## Умные шаблоны заметок

Отдельная фича, не виджет дашборда. Генерирует структуру новой заметки исходя из контекста, а не из статического списка шаблонов.

### Три момента срабатывания

- **При создании** — пользователь вводит название файла, классификатор определяет тип и предлагает структуру
- **По первому абзацу** — через 2 секунды после паузы в печатании LLM читает написанное и предлагает структуру для продолжения
- **По команде** — `/template` в любом месте редактора

### Логика классификатора

Работает в два этапа, чтобы не тратить LLM зря.

**Эвристика (быстро, без LLM)** — набор правил по ключевым словам в названии и тегах. Примеры: `arxiv / paper / статья` → заметка о литературе; `встреча / meeting / 1:1` → заметка о встрече; `ретроспектива / retro` → ретроспектива. Если тип определён — шаблон генерируется без LLM.

**LLM (только если эвристика не уверена)** — получает название файла, первые строки текста и топ-3 похожих заметки из vault по смысловой близости. Ключевое: LLM видит не абстрактный «best practice», а то как сам пользователь структурировал аналогичные заметки раньше. Воспроизводит его паттерн.

### Два режима подачи результата

**Ghost text** — предложенные разделы появляются серым цветом прямо в редакторе ниже текста. `Tab` принимает, `Esc` отклоняет. Не прерывает поток — пользователь может просто продолжить печатать и не заметить.

**Панель выбора** — для сложных типов заметок (научная статья, проект, ретроспектива). Показывает 2–3 варианта структуры с превью, пользователь выбирает. Вставляет скелет с пустыми секциями.

### Обратная связь

Принятые шаблоны фиксируются. Система накапливает информацию о том, какие предложенные структуры пользователь принял, а какие отклонил — и учитывает это при следующих генерациях.

## Поиск противоречий в базе

Инструмент для обнаружения конфликтующих утверждений между заметками. Не виджет дашборда — отдельный раздел или команда.

### Три типа противоречий

**Жёсткое** — одна заметка утверждает X, другая утверждает не-X. Требует явного разрешения.

**Мягкое** — разные акценты или точки зрения на одну тему. Не является ошибкой; может быть нормой или поводом для синтеза.

**Временное** — более старая заметка содержит вывод, который противоречит более новой. Скорее всего старый вывод устарел.

### Логика обнаружения

**Предварительная фильтрация без LLM** — отбирает пары кандидатов по двум критериям: заметки из одного тематического кластера (теги, папка) или заметки с высокой смысловой близостью по эмбеддингам. Пары с максимальной близостью (почти идентичные) исключаются — это дубликаты, не противоречия.

**LLM-анализ пар** — получает пару заметок и определяет: есть ли противоречие, какого типа, какое конкретное утверждение конфликтует с каким, какое действие рекомендуется. Анализируется только топ кандидатов с высоким баллом.

**Кэш** — результаты хранятся и инвалидируются только если одна из двух заметок изменилась.

### Рекомендованные действия

- **Разрешить** — для жёсткого противоречия: открывает обе заметки + чат с LLM для помощи в синтезе. LLM формулирует объединённое утверждение с указанием условий применимости каждой версии.
- **Объединить** — для мягкого: создаёт новую заметку-синтез с обоими взглядами и ссылками на исходные.
- **Архивировать** — для временного: переносит старую заметку в `/Archive/` с тегом `#outdated`.

## Бэклог

Идеи, концептуально проработанные, но не входящие в MVP.

### Анализ базы

**Паттерны мышления** — LLM периодически анализирует все заметки за месяц и выявляет повторяющиеся темы, идеи, подходы. Формулирует наблюдение: «ты часто думаешь про X через призму Y, это встречается в N заметках». Мета-когниция поверх собственного мышления.

**Gaps in knowledge** — LLM находит темы, которые упомянуты в заметках но не раскрыты. Предлагает создать недостающую заметку. Обратная сторона графа: не «что связано», а «что должно быть связано но отсутствует».

### Работа с контентом

**Сократические вопросы** — при сохранении заметки LLM дописывает 2–3 вопроса в конце: «А что если…?», «Чем это отличается от X?», «Какое следствие из этого?». Появляются как ghost text. Форсируют углубление мышления без навязчивости.

**Реструктуризация** — LLM предлагает переработать длинную или хаотичную заметку: разбить на части, добавить заголовки, вынести список. Не автоматическая правка — только предложение с превью.

**Task extraction** — LLM находит в тексте задачи и дедлайны записанные в свободной форме («надо бы разобраться с...», «проверить до пятницы»). Выносит их в трекер задач без ручного труда.

### Навигация

**Маршрут по теме** — если пользователь начинает заметку по незнакомой теме, LLM строит путь изучения из уже существующих заметок vault: «Чтобы разобраться в этом, начни с A → B → C». Персональный curriculum из собственной базы.

**Следующая заметка** — рекомендация что читать или развивать дальше, исходя из текущего контекста и недавней активности.

### Автоматизация

**Авто-теги и категории** — LLM классифицирует новые заметки и предлагает теги и папку. Снижает трение при создании заметок.

**Habit tracker** — трекер привычек из чеклистов в daily note. Виджет с тепловой картой активности.

**Обучение на обратной связи** — для умных шаблонов: система накапливает историю принятых и отклонённых предложений и со временем улучшает точность.

## Ежемесячная карта компетенций

Раз в месяц LLM делает полный просмотр базы и строит таблицу навыков и областей знаний пользователя с краткой оценкой по каждой.

Закрывает слепое пятно всех PKM-систем: база хорошо отвечает на вопрос «что я знаю», но не отвечает на «кто я как специалист» и «как развиваюсь». Карта компетенций делает автопортрет из того что уже написано — без ручного заполнения.

### Ключевое отличие от самооценки

LLM видит не декларацию («я считаю себя экспертом в X»), а доказательства: сколько заметок по теме, насколько они глубокие, растёт ли область. Это честнее ручного скилл-трекера, который быстро устаревает.

### Логика построения

**Шаг 1 — кластеризация.** LLM группирует все заметки по смысловым доменам — не по папкам (которые могут быть хаотичны), а по содержанию. Получается черновая карта областей.

**Шаг 2 — оценка глубины каждого кластера** по нескольким сигналам:
- Объём: количество заметок и суммарный текст
- Глубина: есть ли концептуальные заметки или только поверхностные записи
- Динамика: кластер растёт или стагнирует
- Связность: заметки изолированы или образуют граф между собой

**Шаг 3 — калибровка уровня.** Объём не равен глубине. LLM различает:
- Много заметок с базовыми вопросами → *изучаю*
- Заметки с разбором нюансов, ошибок, трейдоффов → *практикую*
- Заметки где пользователь сам формулирует принципы → *владею*

### Формат таблицы

Четыре колонки: область, уровень (описательный, не числовой), динамика (↑ растёт / → стабильно / ↓ угасает), последняя активность.

### Почему описательный уровень, не числовой

Оценка намеренно словесная — *изучаю, практикую, владею, знаком* — а не цифры 1–10. Цифры создают ложную точность и провоцируют оптимизацию ради показателя. Слова честнее отражают реальный уровень.

### Что можно добавить поверх

**Дельта к прошлому месяцу** — какие области выросли, что появилось новое, что угасает. Мотивирует: видно что реально двигаешься.

**Наблюдения LLM** — необязательный блок под таблицей. Не советы, а наблюдения: «ты активно изучаешь inference, но почти не пишешь про бенчмаркинг — возможно gap». Без навязчивости.

**История по месяцам** — архив предыдущих таблиц. Через полгода-год можно увидеть как менялся профиль компетенций.

## Лента новостей

Отдельная страница приложения — агрегатор новостей по интересующим темам с LLM-фильтрацией и связью с базой знаний.

### Принцип работы

Приложение периодически забирает контент из внешних источников, LLM фильтрует его по профилю пользователя и оставляет только релевантное. Результат — персональная лента без алгоритмов платформ и без ручного отбора.

### Источники по типу доступа

Почти весь нужный контент по теме LLM доступен через RSS или открытые API — без скрейпинга и обхода защит.

**RSS-фиды (нативные, стабильные):**
- Блоги облачных провайдеров: Anthropic, OpenAI, Google DeepMind, Mistral
- Hugging Face Blog — технические посты команды и комьюнити
- Hugging Face Papers — кураторская выборка из arxiv с голосованием
- arxiv категории cs.AI, cs.LG, cs.CL — сырой поток исследований (требует фильтрации)
- Авторские блоги: Simon Willison, Sebastian Raschka (Ahead of AI), The Gradient, Last Week in AI
- GitHub релизы любого репозитория — `github.com/{owner}/{repo}/releases.atom`; покрывает ollama, vLLM, llama.cpp, LM Studio и другие инструменты

**Открытые API (без RSS, но программно доступны):**
- HackerNews API — фильтрация постов по ключевым словам без подписки на весь поток
- Hugging Face API — новые модели в библиотеке, можно опрашивать по дате добавления
- GitHub trending — репозитории по теме за день/неделю

**Источники требующие дополнительного решения:**
- Страницы без RSS (некоторые агрегаторы, Medium за пейволом) — через сторонний сервис типа ScrapingBee или встроенный браузер Electron как крайний случай

### Двухэтапная фильтрация

Сырой поток даже из хороших источников велик — arxiv cs.AI производит десятки тысяч записей в год. Гнать LLM на каждую статью нецелесообразно.

**Этап 1 — keyword-фильтр без LLM.** Быстрая проверка заголовка и тегов по списку ключевых слов пользователя. Отсекает ~70% нерелевантного потока мгновенно.

**Этап 2 — LLM-оценка остатка.** Qwen смотрит на заголовок и первый абзац каждой прошедшей статьи, оценивает релевантность и пишет одно предложение резюме. На вход ~200 токенов, на выход одно предложение — быстро и дёшево по токенам.

### Что LLM делает с отобранными статьями

- Одно предложение резюме: о чём статья и почему может быть интересна
- Кластеризация по темам: не по источнику, а по содержанию
- Связь с vault: «эта статья касается твоей заметки про KV cache» — предлагает ссылку или создание новой заметки
- Оценка новизны: отмечает если тема уже хорошо покрыта в базе

### Производительность на Qwen3 27B

На типичном железе модель выдаёт 15–25 токенов/сек. Без двухэтапной фильтрации полный прогон 200 статей занял бы ~7 минут. С keyword-фильтром на входе остаётся 30–60 статей — прогон сокращается до 1–2 минут.

Оптимальный режим запуска — раз в день в фоне, пока приложение открыто. Не в реальном времени.

### Обновление

Раз в день автоматически — фоновый процесс забирает новые записи из всех источников и прогоняет через фильтры. Пользователь открывает страницу и видит готовую ленту. Кнопка «обновить сейчас» для ручного запуска вне расписания.

## Архитектура кэширования LLM

Независимо от конкретного виджета, кэш работает по единой логике:

- Каждый LLM-виджет хранит результат с временной меткой
- При открытии дашборда проверяется: изменились ли исходные файлы после времени кэша
- Если нет — отдаётся кэш без запроса к модели
- Если да — только затронутые виджеты перегенерируются, остальные остаются
- Тяжёлые операции (противоречия, паттерны мышления) обновляются только вручную или по расписанию раз в сутки

Это позволяет держать страницу отзывчивой даже при медленной локальной модели.
