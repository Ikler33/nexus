# План доработок Nexus — дорожная карта «второй мозг на каждый день»

> Источник: продуктовый аудит `docs/reviews/PRODUCT_AUDIT_2026-06.md` (99 находок, 10 полос) →
> многоагентная проработка реализации (9 стримов × decision-complete → секвенсор → критик автономности,
> 11 агентов, 1.68M токенов, 2026-06-13). Цель этого документа — **исполнять автономно без развилок**:
> каждое продуктовое решение зафиксировано (реестр из 45 решений ниже), срезы упорядочены по
> зависимостям, опасные порядки помечены. Сверка с `docs/BACKLOG.md` — в конце.

> **Главный принцип (мандат 4):** сначала фундамент доверия — нельзя автоматизировать запись/синк/курацию
> поверх не-атомарной записи без guard внешнего изменения. Порядок строгий: atomic-write → хеш-трекинг →
> guard+reload → autosave+flush → история. Временные фичи — на честной таблице `edit_events`, НЕ на `created_at`
> (мандат 5: mtime врёт после git-clone/синка).


## Логика плана

ЕДИНЫЙ ПЛАН: 9 стримов (SAFE, CURATE, CAP, NAV, AIP, EDIT, TASK, POLISH, FLOW) дают ~70 срезов с тяжёлым перекрытием — четыре стрима (SAFE/CAP/EDIT/FLOW) независимо переоткрывают atomic-write, три (CAP/NAV/AIP/TASK) — edit_events, два (SAFE/EDIT) — guard внешнего изменения и историю версий, два (CURATE/CAP) — rename. План ДЕДУПЛИЦИРУЕТ это в общий фундамент и расставляет по зависимостям. Принцип порядка (мандат 4 — СТРОГИЙ): сначала фундамент доверия (нельзя автоматизировать запись/синк/курацию поверх не-атомарной записи без guard), потом всё остальное.

ЛОГИКА ФАЗ. P1 = единственный фундамент сохранности (atomic-write → хеш-трекинг → guard+watcher-reload → debounced-autosave+flush → история backend → история UI). Это канонические SAFE-1..6; их дубли в CAP-0/EDIT-1..4/FLOW-7 ПОГЛОЩАЮТСЯ (исполняем один раз). FLOW-7 поглощён в SAFE-1 + SAFE-4 (flushAllDirty экспортируется отсюда). Пока P1 не влит, нельзя мержить autosave, sync-флаш, link-rewrite, append-в-today, auto-rename — все они пишут чужие/частые файлы.

P2 = ЧЕСТНАЯ ОСЬ ВРЕМЕНИ. Один edit_events (миграция 014) на событиях индексатора — закрывает мандат 5 и разблокирует TASK-календарь, NAV-активность/ресёрфейс, AIP-профиль. КРИТИЧНО: четыре стрима просят «миграцию 014» под разные таблицы — резолвлю коллизию: 014=edit_events, 015=tasks, 016=relation_reasons, 017=versions-index(если понадобится). Схема edit_events — суперсет всех предложений: (id, file_id FK ON DELETE CASCADE, ts, kind, words_delta, words_after, source) — покрывает NAV (words_delta/after), AIP (kind create/modify/delete), CAP/TASK (source app/watcher). Пишется в той же транзакции index_file, ТОЛЬКО при смене хеша (анти-дрожание).

P3 = КУРАЦИЯ (rename/delete/move + link-rewrite + корзина). Зависит от P1 (link-rewrite пишет N чужих файлов атомарно; flush-before-rename). Закрывает BACKLOG #135 (переписывание ссылок) и даёт rename/delete из UI, которых нет.

P4 = ЗАХВАТ И ЯКОРЬ (toast-примитив, daily/journal через lib, шаблоны, фокус курсора, auto-title, inbox-triage, глобальный quick-capture). Toast — общий примитив (POLISH-1 = CAP-3-toast = FLOW-notice), исполняется ОДИН раз в начале P4. Журнал/daily унифицирован: одна папка-дефолт, одна команда note.daily, один lib/daily.ts — три стрима (CAP/TASK/FLOW) сходятся сюда.

P5 = НАВИГАЦИЯ И РЕСЁРФЕЙСИНГ (контент-поиск в сайдбаре/палитре, recents ⌘O, back/forward, курсор/скролл, fuzzy, активность-на-edit_events, on-this-day/random/flashback, mini-calendar). Зависит от P2 (edit_events).

P6 = AI-НАПАРНИК (пин заметки, кликабельные [n], action-row, видимый грунтинг, проактивные инсайты по умолчанию, мосты «разобрать/спросить ИИ», профиль на edit_events, причина связи). Зависит от P1 (append через atomic+guard), P2 (профиль).

P7 = РЕДАКТОР/ПИСЬМО (формат-хоткеи ⌘B/I/K + перенос inline на ⌘J, картинки/embed-рендер, кликабельные чекбоксы, slash-меню блоков, висячие ссылки→создать, трансклюзия, KaTeX-MathML). Фундамент сохранности (EDIT-1..4) уже сделан в P1 — здесь только письмо/рендер.

P8 = КРУПНЫЕ ЭПИКИ (TASK — слой задач: парсер+таблица+панель+чекбоксы+lifecycle целей+LLM-extraction; FLOW-news — пагинация/инфинит/жесты/связь; FLOW-sync — безопасный фоновый commit-then-pull-push+индикатор+авто-резолв). Эти стоят позже: TASK тяжёл (L-срезы) и кормится edit_events(P2)+daily(P4); sync кормится P1+P2.

P9 = ПОЛИРОВКА И ОБНАРУЖИВАЕМОСТЬ (focus-trap/Escape-LIFO, шпаргалка хоткеев, vault-pills чата, welcome-засев, чистка англ-строк+CI-линт, перевод действий на тосты). Завершает «рефлекторность»: пользователь открывает Nexus и всё уже на месте, ничего не молчит.

ОТ «ОСОЗНАННО» К «РЕФЛЕКТОРНО»: P1-P2 убирают страх потери (открываю — доверяю, что не потеряю). P3-P4 делают захват/курацию бесплатными (мысль за <1с, разложить по папкам). P5-P6 делают возврат и AI-помощь рефлексом (⌘O, back, [n]-клик, «разобрать с ИИ»). P7-P9 убирают последнее трение (форматирование/чекбоксы/пустые состояния). Каждый срез = PR (мандат 7: i18n RU/EN + тесты + track/ветка → scripts/test-all.sh без пайпов → PR → squash-мерж на CLEAN, НЕ стекать PR).


## ⚠️ Обязательные поправки критика (применить в P1–P2 до старта потребителей)

Критик автономности поймал три «оно-уже-так-работает» допущения, неверных по факту кода. Они — ЯВНЫЕ
задачи, не предполагаемое поведение:

1. **Per-file событие — это НОВЫЙ код, не существующее.** Сейчас `indexer/events.rs:108` шлёт
   `emit("vault:changed", ())` — без payload, после дрейна всей пачки. Guard SAFE-3 строится на новом
   `vault:file-changed {path, hash}`, который SAFE-3 и заводит (протянуть `rel`+`hash` из `index_file`
   наружу). Guard и watcher-reload — одним срезом, reload не появляется без защиты.
2. **`edit_events` только при смене хеша — НОВЫЙ guard.** `index_file` (`mod.rs:197` `INSERT…ON CONFLICT
   DO UPDATE`) апсертит БЕЗУСЛОВНО, сравнения `old_hash != new_hash` нет. Без явного guard force-rescan/
   реиндекс-при-открытии сгенерит фантомный `modify` на КАЖДЫЙ файл и отравит всю ось времени
   (NAV-8/TASK-5/AIP-9). EVT-1 обязан читать старый `files.hash` ДО апсерта и писать событие (и эмитить
   `file-changed`) только при реальной смене.
3. **Фолбэк активности Home «по mtime» — в ТОМ ЖЕ срезе**, что переключение на `edit_events`. Иначе на
   старом vault таблица пуста → видимая регрессия (пустая активность). Empty-states ресёрфейсинга обязательны.
4. **SAFE-4 — реальный баг не «unconditional dirty=false».** `saveBuffer` (`workspace.ts:181`) делает
   `await writeFile` ДО `set(dirty:false)` — при ошибке throw оставляет `dirty=true`. Настоящий дефект —
   отсутствие `try/catch`+тоста и unhandled-rejection. Критерий переформулирован в срезе SAFE-4.
5. **`closeTab` GC-ит буфер без flush** (`workspace.ts:106`) — реальная потеря правок. Любой срез P4+,
   программно закрывающий вкладки (inbox-triage, news→note, rename-переоткрытие), идёт СТРОГО после SAFE-4.

**Коллизия миграций разрешена:** последняя на диске — `013`; `014=edit_events`, `015=tasks`,
`016=relation_reasons`, `017=versions-index` (если понадобится).

## Первые PR — готовы к коду

### SAFE-1 — Атомарная запись файла (tmp→fsync→rename)

**Что:** Единственная реализация atomic-write для всего проекта (поглощает CAP-0, EDIT-1, FLOW-7-часть). Новый pub fn vault::atomic_write(abs,bytes) через tempfile::NamedTempFile::persist в той же папке + fsync файла + best-effort fsync директории (Unix). write_file заменяет tokio::fs::write на spawn_blocking(atomic_write). При ошибке rename — удалить tmp, вернуть VaultError::Io наверх (тост 'Не удалось сохранить'). Tmp-имя с dot-префиксом уже покрыто watcher::is_ignored (подтверждено watcher/mod.rs:57). Контракт write_file (IPC) не меняется.

**Файлы:** `apps/desktop/src-tauri/src/vault/mod.rs (новый atomic_write), apps/desktop/src-tauri/src/commands/vault.rs:622-629 (write_file)`

**Приёмка:** write_file пишет через tmp+rename; после успеха в каталоге нет tmp-файлов; прерывание между tmp и rename НЕ оставляет усечённый целевой .md (старое содержимое цело — симулировать недоступной целью); watcher не генерирует событий на tmp; существующий тест write_resolve_* и atomic_save_preserves_file_id зелёные.

### SAFE-2 — Хеш контента: write/read возвращают blake3-хеш + команда file_hash

**Что:** Helper vault::content_hash(&[u8])->String (blake3 hex — крейт УЖЕ в Cargo.toml, новой зависимости нет). write_file → возвращает hash записанного. Новая read_file_meta(path)->{content,hash} (read_file оставляем для совместимости). file_hash(path)->Option<String> (None если нет файла). Регистрация в lib.rs generate_handler. Фронт: writeFile возвращает hash, добавить readFileMeta/fileHash; Buffer += baseHash (заполняется при openFile из readFileMeta и saveBuffer из writeFile). mockVault — детерминированный хеш.

**Файлы:** `apps/desktop/src-tauri/src/vault/mod.rs (content_hash), apps/desktop/src-tauri/src/commands/vault.rs (write_file/read_file_meta/file_hash), apps/desktop/src-tauri/src/lib.rs (handlers), apps/desktop/src/lib/tauri-api.ts, apps/desktop/src/stores/workspace.ts (Buffer.baseHash)`

**Приёмка:** content_hash стабилен и различает контент; write_file возвращает хеш = file_hash после записи; file_hash(несуществующий)=None; каждый открытый буфер имеет актуальный baseHash после open и после save (vitest).

### SAFE-3 — Guard внешнего изменения + watcher-reload буфера

**Что:** Backend: indexer/events.rs event_loop при VaultEvent::Upsert после индексации эмитит vault:file-changed {path:rel, hash} (хук как notify), Deleted→vault:file-deleted {path}. ## Реестр решений (45) — исполнять без вопросов

1. **Порядок фундамента (мандат 4)** — СТРОГО: atomic-write(SAFE-1) → хеш-трекинг(SAFE-2) → guard+watcher-reload(SAFE-3) → debounced-autosave+flush(SAFE-4) → история backend(SAFE-5) → история UI(SAFE-6). Autosave НЕ мержить раньше SAFE-1+SAFE-3. Все дубли atomic-write из CAP-0/EDIT-1/FLOW-7 — это SAFE-1 (исполнить один раз).
2. **Атомарная запись** — vault::atomic_write(abs,bytes): tmp `.<basename>.nexus-tmp-<pid>-<rand>` в той же папке → fsync(tmp) → atomic rename → best-effort fsync директории (Unix). ## Детальные срезы по стримам


### Сохранность данных (фундамент доверия) — код SAFE

**Решения стрима:**

- _Порядок срезов — что первым (риск автоматизации потери)?_ → **СТРОГО: SAFE-1 atomic-write → SAFE-2 хеш-трекинг загрузки (фундамент guard'а и истории) → SAFE-3 external-change-guard (баннер) → SAFE-4 debounced-autosave+flush → SAFE-5 backend истории версий → SAFE-6 UI истории. Autosave (SAFE-4) НЕ мержить раньше SAFE-1+SAFE-3; backend истории (SAFE-5) допустимо вести параллельно SAFE-3/4, но autosave включать только когда история уже пишет снапшоты ИЛИ хотя бы snapshot-on-save доступен.** (Поправка критика аудита (строка 139/148): autosave до atomic-write+guard+history = автоматизировать потерю. Хеш-трекинг (SAFE-2) — общий примитив и для guard'а, и для дедупа истории, поэтому выделен в маленький ранний срез.)
- _Как писать файл атомарно на бэкенде?_ → **tmp-в-той-же-папке + fsync(tmp) + atomic rename поверх цели; затем best-effort fsync директории (Unix). Имя tmp: `.<basename>.nexus-tmp-<pid>-<rand>` (dotfile → is_ignored watcher'а и list_dir его не покажут). Реализовать как `vault::atomic_write(abs, bytes)` в vault/mod.rs (новый pub fn), переиспользовать в write_file и в записи снапшотов истории.** (Стандарт write-temp-then-rename (аудит строка 696, эталон Logseq/git). Tmp в той же папке гарантирует rename в пределах одной ФС (на macOS APFS rename атомарен). Dot-префикс ⇒ watcher::is_ignored уже фильтрует (`name.starts_with('.')`), tmp не породит фантомный Upsert. fsync дир — durability против потери rename при отключении питания.)
- _Что если rename упал (нет места/права)?_ → **Удалить tmp (best-effort) и вернуть AppError наверх; фронт показывает тост «Не удалось сохранить» (no silent caps, мандат 3). saveBuffer НЕ сбрасывает dirty при ошибке записи (сейчас сбрасывает безусловно — это баг, чиним в SAFE-4).** (Мандат 3 — любая ошибка видима. Текущий saveBuffer (workspace.ts:185) ставит dirty=false даже не дожидаясь успеха стабильно (await есть, но ошибка пробрасывается и dirty остаётся true только если throw — ок; явно ловим в UI).)

Команда `file_hash(path)` для дешёвой проверки без чтения всего в JS.** (Хеш контента, а не mtime: mtime врёт после git-clone/синка (мандат 5). - _Как фронт узнаёт о внешнем изменении открытого файла?_ → **Бэкенд при Upsert в watcher-петле эмитит новое событие `vault:file-changed` с payload `{path, hash}` (camelCase). Фронт подписывается, и для КАЖДОГО открытого буфера сравнивает hash с baseHash: если расходится — (а) буфер clean → тихо перезагрузить (readFile, обновить doc+baseHash, externalSync-dispatch в Editor — это и есть watcher-reload, которого сейчас нет); (б) буфер dirty И diskHash≠bufferHash → выставить флаг `externalChange` на буфере (баннер). - git_sync (git/mod.rs:411/521 force-checkout) остаётся опасным: SAFE-4 экспортирует flushAllDirtyBuffers, но переход на commit-then-pull и убирание force — отдельный срез стрима sync; до него pull всё ещё может снести незакоммиченную правку, если sync не вызовет flush+чистоту дерева (аудит стр.57/705).
- Перенос истории версий при rename/перемещении заметки не реализован (история привязана к пути) — пока команды rename/delete из UI нет (аудит стр.89). При будущем rename история осиротеет под старым путём (данные целы, но не видны). Связать с rename-срезом.
- Editor.tsx externalSync-dispatch при перезагрузке clean-буфера сбрасывает позицию курсора/скролла на anchor=min(selection,len) — при частых внешних правках (git-синк) возможны прыжки; приемлемо для clean, но стоит проверить на больших файлах.
- onCloseRequested блокирует закрытие при ошибке записи — если ФС стабильно недоступна (vault на отключённом сетевом диске), окно нельзя закрыть штатно; нужен escape-хатч (повторный close в течение N сек → закрыть с предупреждением о потере). Уточнить поведение в SAFE-4.
- content_hash крейт: если ни blake3, ни sha2 не в дереве — добавление зависимости расширяет supply-chain; проверить Cargo.lock в SAFE-2, при отсутствии — взять уже присутствующий хеш (индексатор считает хеши файлов — переиспользовать его функцию вместо новой зависимости).
- Снапшоты истории растут на диске (.nexus/history) — ретенция 50/7дней ограничивает на файл, но в большом vault суммарный объём может быть заметен; нет глобального лимита/GC по общему размеру (можно добавить в backlog).


### CURATE — Курация vault: rename / delete / move + переписывание [[ссылок]] + системная корзина

**Решения стрима:**

- _Куда удалять файлы — системная корзина ОС, .nexus/trash или hard-rm?_ → **Системная корзина ОС через crate `trash` (v5). Никакого hard-rm пользовательских заметок (мандат 4: сохранность). Если `trash::delete` падает (нет корзины на платформе/CI/headless) — НЕ делаем silent fallback на rm: возвращаем типизированную ошибку `AppError::Trash`, фронт показывает тост (мандат 3). Соответствует рекомендации аудита L716-726 «system-trash по умолчанию».** (`.nexus/trash` потребовал бы своей ретенции/GC/восстановления-UI (отдельный эпик), а ОС-корзина даёт восстановление бесплатно и не входит в git/индекс (она вне vault). crate `trash` кроссплатформенный (macOS/Win/Linux-XDG), зависимостей лёгких. Это минимальный безопасный дефолт.)
- _Переписывать [[Old]]→[[New]] у ссылающихся файлов — всегда, по опции, или только wikilinks?_ → **Переписывать по умолчанию ВКЛ (как киллер-фича Obsidian), с тоггл-настройкой `curate.updateLinksOnRename` в prefs (дефолт true). Переписываем wikilink + embed (`[[X]]`, `![[X]]`) и markdown-ссылки на `.md` внутри vault. Сохраняем существующий `|displayText` и `#heading`/`#^block` суффиксы — меняем только targeting-часть. Внешние URL и ссылки на бинарь не трогаем. Каждый затронутый файл переписывается через тот же atomic-write путь (SAFE), что и обычное сохранение.** (Аудит (L89, L196, L719) прямо называет link-aware rename киллер-фичей второго мозга. Тоггл — на случай git-diff-шума. embed+markdown покрывают реальные ссылки Obsidian-vault; сохранение display/heading-суффикса обязательно, иначе мы сломаем подписи ссылок.)
- _Команда rename идёт через ОС fs::rename + watcher, или прямой вызов indexer.rename_file?_ → **Команда `rename_path` делает atomic ОС-операцию (fs::rename файла/каталога) ВНУТРИ vault, затем СИНХРОННО переписывает ссылки в затронутых файлах, и НЕ дёргает indexer напрямую — полагается на watcher (он уже склеивает move в `VaultEvent::Renamed`, watcher:99, и сохраняет file_id, indexer:385). Команда лишь возвращает новый путь + список переписанных файлов для оптимистичного обновления UI; индекс досчитает watcher. Для надёжности (watcher может не склеить на части ФС) команда дополнительно шлёт прямой `index_tx`-сигнал? Нет — НЕ дублируем (двойная индексация). Вместо этого rename_file идемпотентен, и watcher — единственный путь индексации (как сегодня).** (Watcher уже корректно обрабатывает Renamed с сохранением file_id/беклинков. Дублирование через прямой indexer-вызов создаст гонку с watcher-событием на тот же путь. Единый путь индексации = меньше багов. UI обновляется оптимистично по ответу команды, индекс — асинхронно через `vault:changed`.)
- _Как находить файлы со ссылками на переименованный — по таблице links (target_id) или скан всех .md?_ → **По таблице `links`: `SELECT DISTINCT source path FROM links JOIN files WHERE target_id = <renamed file_id>`. Это даёт точный набор файлов-источников без полного скана vault. Затем в КАЖДОМ источнике re-scan текста парсером и замена только тех вхождений, чей нормализованный target резолвится в переименованный file_id (та же `normalize_target` + `resolve_target`, что у индексатора). Висячие ссылки (`target_id IS NULL`) с raw, совпадающим со старым basename/путём, тоже обновляем (опционально, см. риск).** (Таблица links — источник истины беклинков (ADR-004), уже заполнена и проиндексирована. Полный скан vault на каждый rename — O(N) IO зря. Сопоставление через ту же resolve-семантику гарантирует, что мы не тронем ссылку `[[Other]]`, случайно совпавшую по подстроке.)
- _Текстовая замена target — точная по спанам или regex/replace по строке?_ → **Точная: re-парсим источник, для каждого `ParsedLink` получаем байтовый спан целевой части (добавляем `target_span: Range<usize>` в `ParsedLink` — сейчас есть только line_number/context). Заменяем ТОЛЬКО спаны, чьи нормализованные target резолвятся в переименованный файл, заменяя на новый target в минимальной форме (basename-без-.md, если уникален; иначе vault-относительный путь). Идём с конца к началу, чтобы спаны не сдвигались. Никакого str::replace по всему телу (сломал бы совпадения в коде/тексте/других ссылках).** (`normalize_target` уже стрипает `|alias`/`#heading`, поэтому stored target_raw НЕ содержит суффиксов и не годится для in-place замены. Нужен реальный байтовый спан исходного `[[...]]`. Замена по спанам с конца — единственный корректный способ, уважающий код-диапазоны и display-текст.)
- _Что с открытыми буферами (dirty) при rename/delete переименованного/удалённого файла?_ → **Строгий порядок мандата 4. RENAME: если у файла есть dirty-буфер — сначала flush (atomic save) ИЛИ блокируем с тостом «сохраните сначала» (дефолт: авто-flush, т.к. atomic-write уже есть в SAFE; если SAFE ещё не смержен — блок с тостом). После ОС-rename переносим буфер на новый путь в workspace store (новый action `renameBuffer(from,to)`) + переносим вкладки/активную + starred.rename. DELETE: если dirty — подтверждение «несохранённые правки будут потеряны» в диалоге; после удаления closeTab + GC буфера. Файлы-источники, переписанные link-rewrite, если открыты и dirty — НЕ перезаписываем их с диска поверх буфера (guard внешнего изменения — SAFE-зона): показываем баннер «ссылки обновлены на диске».** (Мандат 4 (строгий порядок: atomic-write → guard dirty → ...) запрещает молча терять буфер. Перенос буфера/вкладок при rename — иначе откроется битый путь. Link-rewrite затрагивает ЧУЖИЕ файлы, которые юзер мог открыть и править — нельзя их затирать (это ровно SAFE-guard внешнего изменения).)
- _Поддерживать ли rename/move/delete каталогов, не только файлов?_ → **Да. `delete_path`/`rename_path`/`move_path` принимают и файл, и каталог. Каталог в корзину — целиком (trash удаляет рекурсивно). Rename/move каталога = ОС-rename каталога; индекс досчитает watcher (он рекурсивный и выдаст серию Renamed/Upsert/Deleted по детям). Link-rewrite при rename/move каталога: пересобираем ссылки на ВСЕ .md-потомки (их пути меняют префикс) — собираем file_id потомков по `path LIKE 'olddir/%'` и переписываем источники каждого. Move = частный случай rename (смена родителя).** (Аудит требует «разложить по папкам» (L718) — это move каталогов/файлов. Каталог-rename без обновления ссылок на потомков сломал бы граф массово. ОС-rename каталога атомарен и дешёв; watcher уже рекурсивен.)
- _Move и rename — две команды или одна?_ → **Одна backend-команда `rename_path(from, to)` покрывает оба (rename = смена basename в той же папке; move = смена родителя; комбинация — оба сразу). `to` — vault-относительный путь, резолвится `resolve_vault_path_for_write` (анти-traversal, AC-SEC-1). Отдельной `move_file` нет — UI-аффордансы (F2-rename, DnD-move, «переместить в…») все зовут одну команду. Конфликт цели (файл уже есть) — отказ `AppError::Conflict` с тостом, никакого silent-overwrite.** (Семантически move и rename — одна ОС-операция (rename пути). Одна команда = одна точка валидации/безопасности/тестов. Indexer.rename_file уже обрабатывает оба. Защита от overwrite обязательна (иначе тихая потеря целевого файла).)
- _Каталог-аффордансы создания (New note in folder, New folder) — в этом стриме?_ → **Частично: добавляем `create_dir(path)` команду (нужна для move в новую папку и для контекст-меню «Новая папка»). `createNote(dir)` уже есть в vault store. New note in folder = существующий createNote с непустым dir. Полный шаблон/quick-capture — НЕ этот стрим (это CAPTURE-стрим аудита).** (create_dir — дешёвая предпосылка для move/контекст-меню, логично здесь. Шаблоны/capture — отдельная продуктовая тема (аудит §захват), не курация.)
- _Подтверждение удаления — всегда диалог или тост-undo?_ → **Удаление в корзину — БЕЗ блокирующего диалога по умолчанию (обратимо: корзина ОС), но с тостом «Перемещено в корзину» + кнопка «Открыть корзину» (опционально). ИСКЛЮЧЕНИЕ: если файл dirty — подтверждение (несохранённые правки на диске останутся в корзинной копии, но буфер потеряется). Каталог с >1 файлом — подтверждение с числом файлов. Постоянного удаления из приложения НЕ делаем.** (Аудит L615 рекомендует «обратимое по умолчанию + видимая отмена» (паттерн Linear/Bear). Корзина уже обратима, поэтому диалог на каждый файл — трение. Но dirty-буфер и массовое удаление каталога заслуживают подтверждения.)

**Риски:**

- ЗАВИСИМОСТЬ ОТ SAFE: link-rewrite (CURATE-3) и flush-before-rename (CURATE-6) должны писать через atomic-write SAFE-стрима. Если SAFE ещё не смержен на момент кодинга CURATE — временно использовать существующий tokio::fs::write (как write_file сейчас), но ОБЯЗАТЕЛЬНО переключить на atomic после SAFE (иначе мандат 4 нарушен: переписывание N чужих файлов прямой записью при краше рвёт их). Зафиксировать как явный TODO + тест на atomic после мержа SAFE.
- crate `trash` v5 в headless/sandboxed CI (как dmg-баг §30) и на Linux-серверах без XDG-корзины может падать — тест delete_path должен мягко skip'аться (как live_watcher_delivers_upsert_smoke), а рантайм-ошибка — типизированный тост, не паника. Проверить, что cargo-deny пропускает лицензию trash (MIT/Apache).
- Двойная индексация / гонка: команда rename полагается на watcher для индекса, но если watcher на части ФС не склеит move в Renamed (выдаст Deleted+Created), file_id потеряется и беклинки порвутся. Watcher уже деградирует к delete+create (watcher:154) — в этом случае link-rewrite по старому target_id всё равно отработает ДО реиндекса (мы держим old_file_id из БД до операции), но новый файл получит новый id. Приемлемо для v1; отметить, что rename_file идемпотентен.
- Link-rewrite висячих ссылок (target_id IS NULL), чей raw совпадает со старым basename: их обновление неоднозначно (raw мог указывать на будущий файл). Дефолт CURATE-3 — обновлять ТОЛЬКО резолвящиеся в old_id (target_id совпал). Висячие со совпавшим basename — НЕ трогаем (риск ложной правки). Это сознательно консервативно; может оставить редкий `[[Old]]`, который был висячим. Зафиксировать в доке.
- Каталог-rename с большим числом потомков и источников = много atomic-write + один реиндекс: на крупном vault это заметная пауза. v1 — синхронно в команде (UI ждёт ответ с числом). Если окажется медленно — вынести в фоновую джобу планировщика (есть инфраструктура), но это усложнение отложить до жалобы.
- starred.ts на localStorage не завязан на vault (§355) — rename переносит звезду только для in-app операций (что и делаем). Move/rename снаружи (Finder) звезду не перенесёт — но это вне скоупа CURATE (известный BACKLOG-хвост переезда starred в .nexus/БД).
- target_span в парсере должен корректно считаться для markdown-ссылок `[text](url)` (не только wiki) — спан = диапазон url внутри скобок. Парсер использует pulldown-cmark Range для md (parser:116) — нужно убедиться, что Range даёт байты url, а не всей ссылки; иначе замена сдвинет текст. Юнит-тест (e) в CURATE-3 это ловит.

**Риски:**

- Автосейв (debounced-autosave + flush на blur/close) и история версий (.nexus/history) — НЕ входят в CAP, но мандат #4 ставит их СРАЗУ после atomic-write+guard; CAP-4 auto-rename и CAP-частые-записи безопасны только при atomic-write (CAP-0), но полноценный автосейв нужно вести отдельным стримом (SAFE/DATA) в строгом порядке. Auto-rename до появления автосейва срабатывает только на ручном save — это сознательное ограничение, зафиксировать в BACKLOG.
- Внешнее изменение dirty-буфера (guard по хешу, баннер «Файл изменён на диске») — часть строгого порядка мандата #4 (шаг 2), но это зона стрима сохранности, не CAP. CAP-write пути увеличивают частоту записей → риск гонок с watcher-reload до появления guard; пока полагаемся на то, что capture/daily не открыты как dirty-буфер в момент append.
- tauri-plugin-global-shortcut и второе окно требуют расширения capabilities — есть тест csp_and_capabilities_are_hardened; нужно убедиться, что новый capture.json не вносит fs/shell/http и что окно 'capture' не получает лишних прав. Платформенные грабли (Linux Wayland global-shortcut, macOS accessibility-разрешение) могут потребовать фолбэка/иструкции пользователю.
- edit_events (CAP-1) хук на write_file пишет только app-события; правки, пришедшие через watcher (внешний редактор/git), НЕ дают edit_event → heatmap покажет их только по фолбэку updated_at. ### AIP — «AI как ежедневный напарник»

**Решения стрима:**

- _Как пинить активную заметку в контекст чата — отдельный механизм или поверх существующего RAG?_ → **Чип «Эта заметка» в композере ChatView (toggle, по умолчанию ВКЛ когда открыт .md-файл). Пин не дублирует ретрив, а ГАРАНТИРУЕТ первичность: чанки активной заметки всегда идут как [1..M] перед обычными RAG-хитами. Бэкенд получает `pinned_path: Option<String>`, тянет до PINNED_CHUNKS=4 первых чанков файла (или весь файл если он ≤ ~1500 симв), маркирует их `pinned=true` в SearchHit, ставит первыми и исключает из дедупа с обычным ретривом. center остаётся для граф-ранга. При выключенном чипе поведение прежнее.** (Аудит: «активная заметка почти невидима чату». Гарантия первичности через переупорядочивание дешевле и предсказуемее, чем второй промпт-канал; переиспользует существующий [n]-нумерованный контекст build_rag_messages и кликабельные источники.)
- _Кликабельные цитаты [n] — рендерить как ссылки в markdown или пост-обработкой?_ → **remark-плагин (remarkCitations) превращает текстовые `[n]` в кликабельный `<button data-cite="n">`; маппинг n→source берётся из message.sources/webSources (1-based, как в промпте). Клик: для RAG/memory — openFile(source.path) + (если возможно) скролл к чанку; для web — external.open(url). Неизвестный n (галлюцинация номера) рендерится как обычный текст, не кликается.** (Аудит: «цитаты [n] не кликабельны». Плагин на remark — единственный CSP-safe путь (без dangerouslySetInnerHTML/eval), порядок источников уже совпадает с нумерацией промпта.)

Все действия видимы тостом при успехе/ошибке (мандат 3).** (Аудит прямо перечисляет копировать/повторить/в-заметку/править. «В заметку» обязан идти через atomic-write (мандат 4 — порядок сохранности уже выполнен: append к существующему файлу через временный+rename).)
- _«Спросить ИИ о выделенном» из редактора — это inline-генерация или мост в чат?_ → **Мост в ЧАТ (новый freeform-режим), не inline-вставка. ### FLOW — Новости-интеграция (пагинация/инфинит-скролл, жест «новость → заметка/цитата/в today», связь с заметками базы) + безопасный фоновый sync (опциональный sync-kind: авто-коммит по тишине + периодический pull→push, СТРОГО commit-then-pull с флашем буферов, индикатор удалённых изменений, pull при фокусе окна, авто-разрешение тривиальных конфликтов)

**Решения стрима:**

- _Как именно frontend узнаёт, что есть ещё страница (hasMore)?_ → **Бэкенд НЕ возвращает поле hasMore (NewsPageDto не трогаем сверх добавления). Фронт выводит hasMore эвристикой аудита (строка 525): hasMore = (вернулось РОВНО PAGE_SIZE записей). PAGE_SIZE экспонируем как поле NewsPageDto.pageSize (i64=50), чтобы фронт не хардкодил константу. items аккумулируются в сторе; при смене topic/unreadOnly — сброс на page 0.** (Минимальная правка бэка (одно serde-поле), классический cursor-less infinite-scroll. Точный hasMore (COUNT(*)) дороже и не нужен — лишняя пустая дозагрузка на границе безвредна (вернёт 0 → hasMore=false).)
- _Сентинел IntersectionObserver — на каждую рубрику или один в конце?_ → **Один сентинел-div в самом низу ленты (после последней рубрики), root = scroll-контейнер .page, rootMargin '400px' (предзагрузка до края). Группировка по темам остаётся клиентской по аккумулированному items.** (Лента сгруппирована по топикам, но скролл — единый; один наблюдатель проще и совпадает с эталоном (Reeder/Inoreader). 400px убирает «дёрганье» у края.)
- _Жест «новость → в today»: куда писать, если daily-заметки ещё нет (SAFE/journal не реализован)?_ → **Добавляем news.to_today: дописывает буллет '- [[оригинал]] titleRu — summaryRu' в конец сегодняшней daily-заметки Journal/YYYY-MM-DD.md; если файла нет — создаёт минимальный (только H1 с датой) и дописывает. Путь/формат фиксированы (Journal/, ISO-дата) — НЕ ждём journal-эпик. Через append к markdown (мандат 6), atomic-write (FLOW-7 dep).** («В today» — ядро жеста захвата; завязывать на нереализованный journal-эпик заблокировало бы срез. - _Строгий порядок при фоновом sync, чтобы не потерять данные (force-checkout баг)?_ → **Хендлер SyncHandler::handle строго: (1) запросить флаш всех dirty-буферов фронта и ДОЖДАТЬСЯ (событие sync:flush-request → фронт saveAll → команда sync_flush_ack/таймаут 3с); (2) git_status — если грязно, commit_all (secret-scan; находка → стоп с видимой ошибкой, НЕ коммитим); (3) pull (ff применяется, MergeRequired → СТОП, поднять флаг mergeRequired + событие, НЕ форсить); (4) push. - _Индикатор «есть изменения на удалёнке» — как считать без полного pull?_ → **git_remote_status(): fetch origin (без merge) + сравнить локальный HEAD с FETCH_HEAD через merge_analysis → enum RemoteStatus { UpToDate, Behind{n}, Diverged }. Реюз merge_analysis уже есть в pull().)
- _Pull при фокусе окна — что именно делает?_ → **Слушатель Tauri onFocusChanged(focused=true) на фронте: если remote настроен И autoSync включён И прошло ≥ FOCUS_PULL_THROTTLE(60с) — ставит джобу kind 'sync' (enqueue, дедуп has_ready_job). НЕ синхронный pull в UI-потоке — через планировщик (единый безопасный путь commit-then-pull). **Риски:**

- Зависимость от SAFE: FLOW-7 закрывает лишь МИНИМУМ (atomic-write+flushAllDirty+sync:flush). Полная сохранность (guard внешнего изменения dirty-буфера в Editor.tsx:126-140, debounced-autosave, .nexus/history) — отдельный SAFE-стрим; до него фоновый sync безопасен по данным НА ДИСКЕ, но несохранённый буфер, перетёртый external-sync-dispatch без dirty-guard, остаётся отдельной дырой (вне FLOW). Порядок мандата 4 соблюдён: autosave/история — НЕ в этом стриме.
- news_to_today/quote пишут в Journal/YYYY-MM-DD.md по ЗАШИТОМУ пути/формату до journal-эпика. Если journal-эпик выберет другой путь/формат (настраиваемый каталог) — потребуется миграция/реконсиляция уже созданных файлов. Зафиксировано как дефолт; журналу унаследовать тот же путь.
- Авто-резолв append-only (FLOW-11) опирается на эвристику «base — общий префикс ours и theirs»: при перемешанных правках в середине daily эвристика вернёт None (безопасно, уйдёт в ручной resolver), но НЕ покрывает все «ощущаемо тривиальные» случаи. Консервативно по дизайну; калибровка — по реальным конфликтам владельца.
- git-фетч идёт через git2/vendored-openssl с токеном из keychain, а НЕ через net::GuardedClient — это отдельный egress-канал вне CI-линта check-egress. Нужно явно отметить исключение в net.md/линте, иначе будущий аудит сочтёт это дырой egress-контроля (или обернуть remote-хост в проверку политики перед fetch).
- Фоновый pull при фокусе + recurring pull могут совпасть с ручным sync → конкуренция за git_lock (сериализует, не повреждает) и дубль-джобы (has_ready_job дедуп). Риск low, но при медленной сети фокус-флаппинг + 60с-троттл стоит проверить на живом remote (FLOW-12).
- news_related считает эмбеддинг+поиск on-demand при каждом открытии reader без кэша — на слабом эмбеддере добавит задержку открытия. Кэш (по item.id, инвалидация по mtime индекса) отложен; триггер — жалоба на лаг reader.

## Блокеры автономности (требуют владельца / ОС / секрета)

Эти срезы НЕ блокируют автономное исполнение ядра P1–P7; вынесены отдельно с фолбэками:

- tauri-plugin-global-shortcut отсутствует + macOS Accessibility consent для CmdOrCtrl+Shift+Space (CAP-6) — ОС-разрешение даёт ТОЛЬКО владелец вручную; автономный smoke невозможен.
- crate trash отсутствует в дереве; добавление + cargo-deny лицензионная проверка (CURATE-1) — при несовместимой лицензии нужен владельческий выбор fallback.
- git-sync (FLOW-8..12): keychain-токен для push к реальному remote — секрет владельца; autoSync ВЫКЛ по умолчанию (consent-gated). Live-smoke push/pull (FLOW-12) требует реального remote и токена — недоступно агенту без владельца.

- Новая capability capture.json + расширение теста csp_and_capabilities_are_hardened (CAP-6) — изменение security-поверхности; по мандату 2 минимальные capability ок, но сам факт нового capability обычно требует владельческой отмашки на security-surface.
- LLM-зависимые срезы (AIP-5 проактивные инсайты по умолчанию, TASK-7 extraction, AIP-9 профиль, AIP-10 причина-связи) гоняют фоновую нагрузку на 192.168.0.31; eval-гейты (live_eval) требуют живого сервера — если сервер недоступен/слетел после ребута, гейт не пройти автономно.

## Вердикт критика автономности


УСЛОВНО ГОТОВ к автономному исполнению P1-P2, с ОБЯЗАТЕЛЬНЫМИ правками фундамент-срезов ПЕРЕД стартом; P3+ имеют owner/ОС-блокеры. Поправить в первую очередь (по убыванию критичности): (1) SAFE-3 строит guard на НЕсуществующем событии — текущий хук это параметрless emit('vault:changed',()) (events.rs:106-108), без path/hash, после дрейна пачки; index_file ничего не возвращает. Сначала срез 'протянуть rel+hash из index_file в per-file emit vault:file-changed', и ТОЛЬКО потом SAFE-3/EVT-1; иначе guard и edit_events молча no-op, а autosave (SAFE-4) ляжет поверх неработающего guard = прямое нарушение мандата 4. (2) EVT-1 'только при смене хеша' НЕ обеспечен: files-upsert безусловен (mod.rs:198), сравнения old_hash!=new_hash нет — без явного guard force-rescan/реиндекс-при-открытии сгенерит фантомный modify на каждый файл, отравив всю честную ось времени (NAV-8/TASK-5/AIP-9). (3) Фолбэк 'по mtime' для активности Home обязан влиться В ТОМ ЖЕ срезе, что переключение на edit_events, иначе видимая регрессия на старых vault. (4) SAFE-4 acceptance мис-специфицирован: saveBuffer НЕ делает 'unconditional dirty=false' — await writeFile бросает ДО set(dirty:false), так что dirty остаётся true; реальный баг — отсутствие try/catch+тоста и unhandled-rejection, не silent dirty-clear. Переформулировать критерий. (5) closeTab GC-ит буфер без flush (подтверждено) — реальный data-loss; любой стрим, закрывающий вкладки программно, строго после SAFE-4. Блокеры автономности вне кода: global-shortcut plugin + macOS Accessibility (CAP-6), crate trash + cargo-deny (CURATE-1), git keychain-токен + реальный remote (FLOW-sync), MathML/трансклюзия вебвью-смоук и новый capture.json capability — все owner/ОС/секрет-gated. Коллизия миграций (014=edit_events/015=tasks/016=relation_reasons) разрешена чисто; 013 — последняя, 014 свободна. Грунтинг плана в основном точен (is_ignored, blake3/tempfile/usearch, отсутствие onCloseRequested/Editor.focus подтверждены), но три 'уже-так-работает' допущения (per-file emit, hash-change-only upsert, unconditional dirty-clear) фактически неверны и должны стать ЯВНЫМИ задачами, а не предполагаемым поведением."}


**Нерешённые развилки (под наблюдением):**

- CURATE-3 'переписывание [[ссылок]]' зависит от добавления target_span:Range в ParsedLink (parser/mod.rs:33 НЕ имеет байт-спанов, только line_number/context). Это net-new парсер-работа в горячем index-пути; план объявляет байтовую замену 'решённой', но реализации спанов нет. Развилка 'спаны или regex-по-строке' формально закрыта в пользу спанов, но цена (рефактор parser + реиндекс всех ссылок) не оценена — агент может застрять на выборе, ломать ли существующий контракт ParsedLink.
- CAP-6 глобальный quick-capture: tauri-plugin-global-shortcut ОТСУТСТВУЕТ в Cargo.toml (подтверждено grep). Дефолтный хоткей CmdOrCtrl+Shift+Space на macOS требует Accessibility-разрешения ОС — это owner/ОС-gated, агент не сможет验证 автономно (план сам помечает 'платформенные грабли → фолбэк-инструкция', т.е. остаётся ручной шаг владельца).
- CURATE-1 удаление в системную корзину: crate trash ОТСУТСТВУЕТ в дереве (подтверждено grep — есть blake3/tempfile/usearch, нет trash). Добавление зависимости + cargo-deny проверка лицензии = развилка, которая при неподходящей лицензии (план сам требует 'cargo-deny проверить') заставит спросить владельца про fallback-стратегию.
- Перенос истории версий/starred при rename явно оставлен 'хвостом BACKLOG' — но CURATE-2/4 дают rename из UI РАНЬШЕ, чем решён перенос истории. Пользователь переименует файл и тихо осиротит свою .nexus/history — решение 'данные целы, история осиротеет' принято, но это product-развилка (молчаливая потеря доступа к истории), которую владелец может не одобрить.
- Авто-резолв конфликтов в FLOW-11 ('только add/none + append-only конкатенация уникальных строк') — граница 'тривиального' конфликта субъективна; реальный merge заметки с двух машин почти всегда content-conflict, не add/none. Агент упрётся в 'что считать тривиальным' на первом же непустом diff и/или владелец должен подтвердить consent для autoSync (план: autoSync ВЫКЛ по умолчанию — значит фича недоступна без явного владельческого включения).

**Опасные порядки исполнения:**

- edit_events 'только при смене хеша' (EVT-1) технически НЕ обеспечен текущим index_file: на force-реиндексе (mod.rs:158-166 mtime-шорткат пропускается) хеш считается, но upsert files (mod.rs:198) безусловный — НЕТ сравнения hash != files.hash перед записью. Если EVT-1 пишет edit_event в той же транзакции без явного 'old_hash != new_hash' guard, force-rescan/реиндекс при открытии vault сгенерирует ФАНТОМНЫЕ modify-события на КАЖДЫЙ файл. Анти-дрожание надо реализовать НОВЫМ кодом, а не полагаться на существующее поведение.
- SAFE-3 предполагает событие vault:file-changed {path,hash} от индексатора, но реальный хук — параметрless notify()→emit('vault:changed',()) (events.rs:106-108), без path и hash, и срабатывает ПОСЛЕ дрейна всей пачки. Guard внешнего изменения, построенный на несуществующем per-file событии, не сработает; нужно сперва протянуть rel+hash из index_file наружу (index_file ничего не возвращает). Порядок 'reload+guard одним срезом' верен, но фундамент-событие надо построить ДО SAFE-3, иначе guard молча no-op и autosave (SAFE-4) встанет поверх неработающего guard — ровно нарушение мандата 4.
- closeTab (workspace.ts:106-127) GC-ит буфер БЕЗ flush — подтверждено. Если SAFE-4 (flush-before-GC) не влит, а любой срез P4+ начинает программно закрывать вкладки (inbox-triage, news→note, rename переоткрытие) — теряются dirty-правки. Любой стрим, закрывающий вкладки, ДОЛЖЕН идти строго после SAFE-4.
- rename-rewrite (CURATE-3) пишет N чужих файлов; если он влит до того как atomic_write реально подключён к link-rewrite пути (а не только к write_file), частичный сбой на середине пачки оставит часть ссылок переписанными, часть нет, без атомарности на уровне набора. atomic_write — пофайловый, не транзакция набора; 'строгий порядок' не покрывает batch-atomicity link-rewrite (нет отката переписанных файлов при сбое на 5-м из 10).
- on-this-day/heatmap/календарь (NAV-8/9, TASK-5) на edit_events — корректно по мандату 5, НО до накопления истории таблица пуста, а активность Home переключается с updated_at на edit_events. Если переключение (decisions: 'Активность Home с updated_at на edit_events') влито до seed/backfill, существующие vault'ы покажут ПУСТУЮ активность пока не накопится edit_events — фолбэк 'по mtime' обязателен и должен влиться В ТОМ ЖЕ срезе, иначе регрессия видимых данных.
