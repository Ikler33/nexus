# Стратегия комплексного автоматизированного тестирования Nexus

> Источник истины по требованиям: `docs/acceptance/ACCEPTANCE.md` (AC-…) и `docs/architecture/ARCHITECTURE.md` (v1.1). Этот документ описывает **как** мы проверяем каждую фичу и функцию автоматически, чтобы агент мог самостоятельно гонять полный набор тестов до PR, а CI подтверждал результат и ловил регрессии без участия человека.
> Числовые пороги (coverage, eval recall/nDCG/MRR, perf) — placeholder’ы из `ACCEPTANCE.md`, фиксируются как baseline на реальных данных; здесь задаются механизмы и стартовые значения.

## 1. Терминология (чтобы закрыть «я не знаю как это называется»)

То, что вы описываете словами «тестирование на каждую фичу и функцию, прогон автоматом при релизе фичи, ловить большинство багов при разработке» — в индустрии это сумма нескольких устоявшихся практик. Короткий словарь, дальше каждый термин раскрыт по нашему стеку:

- **Test pyramid (пирамида тестов)** — модель распределения тестов по слоям: много дешёвых и быстрых **unit**-тестов внизу, меньше **integration**, совсем немного дорогих **E2E** наверху. Анти-паттерн «перевёрнутая пирамида» / «ice-cream cone» — когда почти всё на E2E: медленно и хрупко.
- **Unit / integration / E2E (end-to-end)** — три слоя пирамиды. Unit: одна функция/модуль изолированно. Integration: несколько модулей вместе через реальную границу (БД, файловая система, Tauri-команда). E2E: всё приложение целиком как пользователь — клики по UI, реальный backend.
- **Smoke test** — узкий быстрый прогон «приложение вообще запускается и базовый happy-path работает». Первое, что гоняют; если упал — остальное не запускают.
- **Regression suite (регрессионный набор)** — накопленный набор тестов, который защищает уже работающее поведение от поломки будущими изменениями. Каждый закрытый баг → тест в этот набор, чтобы баг не вернулся.
- **Continuous testing (непрерывное тестирование)** — тесты гоняются автоматически на каждое изменение (локально у разработчика/агента и в CI), а не «отдельной фазой QA в конце».
- **CI quality gates / merge gates (гейты качества / гейты слияния)** — автоматические проверки в CI, которые **блокируют** merge при провале (типичный механизм — required status checks в GitHub branch protection). «Quality gate» — общий термин (любая проверка), «merge gate» — тот же гейт в роли блокировки слияния.
- **Traceability matrix (матрица прослеживаемости) «требование ↔ тест»** — таблица/файл, где каждое требование (у нас — каждый `AC-…`) связано с конкретными ID тестов, доказывающих его. Отвечает на вопрос «эта фича вообще покрыта тестом?».
- **Coverage (покрытие кода) + coverage gate + ratcheting** — доля кода, исполняемого тестами. «Gate» — порог, ниже которого CI падает. «Ratchet/трещотка» — политика «порог может только расти»: новый код не имеет права снижать покрытие.
- **Visual regression testing (визуальная регрессия)** — скриншот-тесты: эталонный снимок UI сравнивается пиксельно/перцептивно с текущим; ловит непреднамеренные изменения вёрстки.
- **Golden test / eval harness** — у нас RAG-eval: фиксированный «золотой» набор `вопрос → ожидаемые файлы` + метрики (recall/nDCG/MRR) против зафиксированного **baseline**. Это регрессионный гейт качества для недетерминированной части (поиск/ранжирование).
- **Flaky test / quarantine (флоки / карантин)** — нестабильный тест (то зелёный, то красный без изменения кода). «Карантин» — изоляция таких тестов из блокирующего гейта с отдельным отчётом, чтобы они не блокировали merge и не приучали игнорировать красный CI.
- **No silent caps / no silent skips** — правило: пропущенные/`#[ignore]`/`skip`/карантинные тесты должны **явно репортиться** числом, а не «молча» делать прогон зелёным. «Зелёный, потому что ничего не запустилось» — худший исход.

Сжатый ответ владельцу: то, что вы хотите, называется **test pyramid + continuous testing с CI merge gates и traceability-матрицей фича↔тест**; «прогон автоматом при релизе фичи» = **CI quality gates**; «ловить большинство багов при разработке» = плотный нижний слой пирамиды + **regression suite** + **coverage ratchet**.

Фактический baseline зафиксирован (bge-m3 @ :8083, k=8, 10 кейсов: recall@8=1.000, nDCG@8=0.883, MRR=0.848). Описан в `docs/dev/eval.md`. **AC-EVAL-1/2/6 фактически закрыты.**

**Perf-бенч** (`eval::tests::bench_index_scale`, `#[ignore]`): синтетический vault `NEXUS_BENCH_FILES` нот, индексация с живым bge-m3, замер индексация/поиск/граф + проекция на 50k. Реальные числа в `docs/dev/perf.md`.

**CI** (`.github/workflows/ci.yml`), 2 джобы, гейтит уже сейчас:
- `frontend` (ubuntu): `pnpm install --frozen-lockfile` → `typecheck` → `lint` → `test` → `build`.
- `rust` (working-dir `src-tauri`): `fmt --check` → `build --locked` → `clippy --all-targets -D warnings` → `test --locked`. Матрица ОС умная: на PR ubuntu+windows (экономия минут), на push в main — +macos. `concurrency` с `cancel-in-progress`.

### Чего НЕТ (честные пробелы)

1. **E2E всего приложения отсутствует полностью.** Нет `tauri-driver`/WebdriverIO/Playwright, нет `test:e2e`-скрипта, нет `.e2e.` файлов. Vitest гоняет React в jsdom **без реального Rust-backend и без WebView** — связка «UI → Tauri IPC → Rust-команда → rusqlite/usearch/git2» целиком как у пользователя нигде не проверяется автоматически. Frontend-тесты ходят в **mock-backend**, а не в настоящие команды.
2. **Coverage-гейта нет вообще.** Нет `cargo-llvm-cov`/tarpaulin, в `vitest.config.ts` нет блока `coverage`, нет Codecov/порога/ratchet. `AC-Q-2` (старт 70% на indexer/chunker/RRF/broker/watcher) **декларирован, но не измеряется и не гейтится**.
3. **Визуальной регрессии нет.** `AC-PR-4` (states empty/loading/streaming/error, a11y, контраст) проверяется глазами, не скриншотами.
4. **Traceability «фича/AC ↔ тест» не существует как артефакт.** `ACCEPTANCE.md` даёт ID `AC-…`, но **обратной связи нет**: нельзя автоматически сказать, какие AC покрыты, а какие — нет. Хуже: **commit-сообщения сейчас НЕ ссылаются на AC-…** (проверено по `git log` — ни одного `AC-` за последние 30 коммитов, и репозиторий на текущей машине не инициализирован как git). То есть «привязка к AC в commit-месседжах» — это **намерение**, а не факт; стратегия должна её **установить**, а не «продолжить».
5. **Live-гейты (eval + perf) не идут в CI.** Оба под `#[ignore]` и требуют embedding-сервер :8083. В CI исполняется только математика метрик. Значит `AC-EVAL-3` и `AC-PERF-*` **в автоматике на каждый PR не гейтятся** — нужен либо кэш эмбеддингов (фикстура векторов, уже в BACKLOG eval.md), либо сервис-контейнер.
6. **Нет «no silent caps»-учёта.** `#[ignore]`-тесты и любые `skip` молча выпадают из зелёного прогона; число пропущенных нигде не репортится как гейт.
7. **Нет единой команды «прогнать всё».** `pnpm test` (Vitest) и `cargo test` запускаются раздельно; live-eval/perf — руками с сервером. Агент не имеет одной точки входа «полный прогон перед PR».

Вывод аудита: **нижний слой пирамиды (unit) — зрелый и местами образцовый (security, eval)**; **средний слой частично есть** (rusqlite/usearch/git2 щупаются внутри unit-тестов через `tempfile`), но **не выделен**; **верхний слой (E2E) и инфраструктура автономности (coverage-гейт, traceability, единый прогон, CI-eval/perf) — отсутствуют**. Именно их строит этот документ.

## 3. Целевая пирамида под наш стек

```
                ╱╲   E2E (tauri-driver + WebdriverIO)        ~5%   ← смоук критичных путей
               ╱──╲  Visual regression (опц., Ф4)            ~3%
              ╱────╲ Integration (Tauri cmd ↔ rusqlite/usearch/git2)  ~20%
             ╱──────╲ Unit (cargo test / Vitest+RTL)          ~70%   ← основная масса
            ────────── вне пирамиды, но в CI-наборе:
                       RAG-eval (golden+baseline)  ·  Perf-бенч (проекция на 50k)
```

Доли — целевой ориентир по **количеству** тестов (не по времени). Дорогие слои держим узкими намеренно.

### Слой 1 — Unit (~70%). Инструменты: `cargo test`, Vitest 3 + Testing Library

- **Покрывает:** чистые функции и изолированные модули. Rust — математика метрик (`eval`), парсер wikilink/tag, чанкер, RRF-слияние, glob/scope-логика permission, layout-расчёты. Frontend — Zustand-сторы, `lib/*`, i18n-плюрализация/ключи, рендер компонентов в jsdom, состояния (empty/loading/error) на уровне компонента.
- **Что усилить:** довести до правила «каждая новая чистая функция/редьюсер/команда → unit-тест» (формализуется в §5 как coverage-ratchet). База уже плотная (116 + 85).
- **Скорость:** секунды; гоняется на каждое сохранение/каждый PR.

### Слой 2 — Integration (~20%). Инструменты: `cargo test` в выделенном `src-tauri/tests/`, Vitest-контракт-тесты mock↔real

Сейчас integration «спрятан» внутри `mod tests` через `tempfile`. **Выделяем явно**, потому что это второй по ценности слой для отлова регрессий без E2E-цены:

- **Rust integration (`src-tauri/tests/*.rs` — отдельный крейт, чёрный ящик публичного API `nexus_desktop_lib`):**
  - **Tauri-команды целиком**: вызвать команду (`vault.*`, `git.*`, индексация) с реальными rusqlite + usearch + git2 на `tempfile`-vault, проверить эффект в БД/индексе/репозитории. Закрывает AC-Б7 (write-actor, `SQLITE_BUSY` под N-конкуренцией), AC-Б8 (FTS-триггеры, чистка векторов при удалении), AC-Б9 (atomic-save tmp+rename, `file_id` стабилен, watcher `is_ignored`).
  - **rusqlite-миграции**: применить миграцию на чистой и на «старой» БД; пересоздание usearch при смене `embedder.dim()` (AC-Б5-2, AC-PR-3).
  - **git2**: commit без `.nexus/plugins/**` (AC-Б3-1), pull→needs-review (AC-Б3-2), secret-scan, sync-lock; реальный временный репозиторий.
  - **usearch**: размерность = `dim()`, отказ на вектор иной длины (AC-Б5-1), удаление векторов файла без дублей при реиндексации (AC-Б4-2, AC-Б8-2).
- **Frontend integration (Vitest):** контракт-тесты «mock-backend ↔ форма реальных команд» — гарантия, что `lib/mock/*` остаётся изоморфным реальным Tauri-командам (тот же shape ответа). Это **страховка детерминизма**: пока mock и real совпадают по контракту, E2E можно держать узким.
- **Скорость:** секунды–десятки секунд; на каждый PR.

> **Важно для CI-расходов:** vendored `git2` (libgit2+openssl) и `usearch` уже заметно удлиняют сборку Rust (отмечено прямо в `Cargo.toml`). Integration-слой это не ухудшает (та же сборка), но E2E добавит сборку **release-бинаря приложения** — поэтому E2E держим на pre-release/ночь, не на каждый PR (см. §5).

### Слой 3 — E2E (~5%). Инструмент: `tauri-driver` + WebdriverIO (официальный путь Tauri 2)

- **Покрывает:** критичные пользовательские сценарии целиком, real backend + real WebView: открыть vault → дерево рендерится → открыть заметку → редактор принимает ввод → `[[wikilink]]` автокомплит/клик → FTS находит → беклинки показаны → command palette → вкладки/сплиты сохраняют dirty-буфер (AC-DOD-Ф0 целиком). Для Ф1: «Что я знаю о X?» → ответ со стримингом и источниками (AC-Б10).
- **Реализация:** `tauri-driver` — официальная обёртка Tauri, поднимающая WebDriver-сервер поверх нативного webview-драйвера платформы; сверху любой WebDriver-клиент, рекомендованный — **WebdriverIO**. Запуск в CI на Linux — через `xvfb` (headless). Это [официально документированный путь Tauri 2 для E2E](https://v2.tauri.app/develop/tests/webdriver/).
- **Ограничение, которое надо принять честно:**
  - `tauri-driver` опирается на нативные WebDriver платформы: **Linux — `WebKitWebDriver` (WebKitGTK)**, **Windows — `Edge WebDriver` (WebView2)**. **macOS — НЕ поддерживается**: у WKWebView нет публичного WebDriver-сервера, `tauri-driver` под macOS работать не будет.
  - Практический вывод: **E2E гоняем на Linux (основной) + опц. Windows**; **на macOS E2E недоступен**, и его пробел закрываем (а) integration-слоем (real rusqlite/usearch/git2/команды — это и так кроссплатформенно через `cargo test` на всех трёх ОС в CI), (б) запуском полной Rust-матрицы на push в main (уже настроено — там +macos). Так macOS-специфика ловится на уровне команд/ФС, а E2E-«склейка с WebView» проверяется на Linux/Windows, где WebView-движок всё равно другой (WebKitGTK/WebView2), т.е. E2E на одной ОС не даёт ложной уверенности про другую — это нормально и типично для Tauri.
  - **Альтернатива/дополнение, не замена:** держать тонкий **smoke-харнесс на Vitest+RTL с real-shape mock-backend** (у нас уже есть `lib/mock/*`) как «E2E-lite» для macOS-разработки агента — быстрый, детерминированный, без WebDriver. Полноценный нативный E2E — только Linux/Windows CI.
- **Скорость:** десятки секунд–минуты (нужна сборка приложения). Поэтому **узко** и **не на каждый PR** — на pre-release и в ночном прогоне.

### Слой 4 — Визуальная регрессия (опц., ~3%, Фаза 4)

- **Покрывает:** AC-PR-4 (empty/loading/streaming/error/offline, контраст), тема light/dark (AC-DOD-Ф4), отсутствие случайных регрессий вёрстки.
- **Инструмент:** на старте — **WebdriverIO visual service** (скриншоты в рамках того же E2E-стека, без второго раннера) либо отдельный **Playwright `toHaveScreenshot`** на статически собранном фронте (`vite build` → отдать `dist/` без Tauri) для скорости и кроссплатформенности (Playwright Chromium стабилен на всех ОС, в т.ч. macOS). Рекомендация: **Playwright поверх собранного фронта** для визуалки (детерминированно, быстро, macOS-friendly), а нативный E2E оставить WebdriverIO+tauri-driver.
- Эталоны — в репозитории; перцептивный порог, не побитовый. Откладываем до Ф4 (стабилизации UI), чтобы не переснимать эталоны на каждый редизайн.

### Вне пирамиды, но в наборе

- **RAG-eval** (есть): golden+baseline, recall@8/nDCG@8/MRR. Гейт качества недетерминированной части. CI-вариант — на кэшированных эмбеддингах (см. §5/§7).
- **Perf-бенч** (есть): `bench_index_scale`, проекция на 50k, пороги AC-PERF-*. Регресс-гейт на pre-release.

## 4. Traceability «фича/AC ↔ тесты»

Цель: на любой `AC-…` из `ACCEPTANCE.md` автоматически ответить «какими тестами доказан» и **завалить CI, если AC не покрыт ни одним тестом**. Это и есть «тест на каждую фичу/функцию» в проверяемой форме.

### Формат: машиночитаемая матрица + конвенция тегирования в тестах

Один источник истины — файл **`docs/dev/traceability.yml`** (машиночитаемый, ревьюится в PR, рендерится в таблицу скриптом). Структура:

```yaml
# docs/dev/traceability.yml — фича/AC ↔ тесты. Живёт через CI-проверку покрытия AC.
AC-Б2-1:
  feature: "Capability-broker: identity по порту (confused-deputy)"
  phase: Ф2
  tests:
    - rust: plugin/broker.rs::identity_is_per_token_confused_deputy
```

Почему YAML-файл, а не «только теги в тестах»: нужен **обзор сверху** («какие AC ещё `pending`») и нужно покрывать AC, которые доказываются **связкой нескольких тестов разных слоёв** (один тег в одном файле этого не выразит). Теги в тестах — вторичны и опциональны (комментарий `// AC-Б2-1` рядом с `fn`), матрица — первична.

### Как держать живой (CI-проверка)

Скрипт **`scripts/check-traceability.mjs`** (Node, гоняется в CI-джобе `traceability`):
1. **Парсит `ACCEPTANCE.md`** → множество всех `AC-…` (regex `AC-[A-ZА-Я0-9-]+`).
2. **Парсит `traceability.yml`** → какие AC заявлены и с какими тестами.
3. **Проверяет существование тестов**: для каждого `rust:`/`front:`/`e2e:` грепом по `src-tauri/src`, `apps/desktop/src`, `e2e/specs` убеждается, что функция/`it()`/spec с таким именем реально существует (ловит «протух тест, а матрица ссылается»).
4. **Гейты:**
   - **Каждый AC из ACCEPTANCE.md обязан присутствовать в матрице.** Нет записи → CI **падает** («AC-XXX не имеет traceability-записи»). Это и есть «тест на каждую фичу»: новый AC нельзя смержить без записи о тесте.
   - Запись со `status: pending` или `covered-live-only` **допустима, но печатается в отчёт явно** (раздел «No silent caps», §5) — чтобы непокрытые AC были видны числом, а не забывались.
   - Ссылка на несуществующий тест → CI **падает**.
5. **Рендерит** `docs/dev/traceability.md` (человекочитаемая таблица AC × тесты × статус) — артефактом сборки, чтобы владелец видел картину покрытия фич без чтения YAML.

### Привязка к commit-сообщениям (устанавливаем, т.к. сейчас её нет)

- Конвенция коммита: тело содержит `Closes AC-…` / `Refs AC-…`. Лёгкий **commit-msg/PR-чек** (необязательный, warning-only на старте): если PR трогает код фичи, но в diff нет ни строки `AC-` и нет изменения `traceability.yml` — предупредить ревьюера. Не делаем это жёстким блокером сразу (чтобы не ломать поток), но включаем в шаблон PR.
- `AC-PR-1` («изменение покрыто тестом(ами) со ссылкой на AC») становится **проверяемым автоматически** именно через эту матрицу, а не «на совесть».

## 5. CI quality gates

Принцип двух эшелонов: **быстрый на каждый PR** (минуты, детерминированный, без сети/серверов) и **полный на pre-release/ночь** (E2E + live-eval + perf + coverage-тренд). Реализуется как required status checks в branch protection на `main`.

### На КАЖДЫЙ PR (быстрый эшелон, расширяет текущий `ci.yml`)

| Джоба | Шаги | Гейтит AC |
|---|---|---|
| `frontend` (есть) | `typecheck` → `lint` → `vitest run` → `build` | AC-Q-1/3, AC-I18N-1 (линтер ключей) |
| `rust` (есть) | `fmt --check` → `build --locked` → `clippy -D warnings` → `cargo test --locked` (матрица ОС как сейчас) | AC-Q-1/3, и весь нижний+integration слой |
| **`coverage` (новая)** | `cargo llvm-cov` + `vitest --coverage` → проверка порога и **ratchet** (не ниже baseline) | **AC-Q-2** |
| **`traceability` (новая)** | `node scripts/check-traceability.mjs` | **AC-PR-1**, покрытие всех AC |
| **`lint-keywords` (новая, дешёвая)** | греп «снятых решений» (sqlite-vec как ANN, petgraph-источник, WASM-логика, `currentFile`) вне docs | **AC-Q-6** |
| `eval-math` (часть `rust`) | unit-математика метрик + парс golden/baseline (уже в `cargo test`) | часть AC-EVAL |

Этот эшелон **детерминирован**: никаких живых LLM/embedding-серверов, никакой сети (`--frozen-lockfile`, моки). Цель — зелёный за единицы минут, чтобы агент гонял его перед каждым PR.

### На pre-release / ночной прогон (полный эшелон)

Отдельный workflow `release.yml` (триггеры: тег `v*`, `workflow_dispatch`, ночной `schedule`):

| Джоба | Что | Гейтит |
|---|---|---|
| **`e2e`** | сборка приложения + `tauri-driver`+WebdriverIO на **Linux (xvfb)** и **Windows**; смоук критичных путей | AC-DOD-Ф0/Ф1 (UI-склейка) |
| **`eval-live`** | поднять embedding-сервер как service-container (или кэш-фикстура векторов) → `eval::tests::live_eval_meets_baseline` | **AC-EVAL-3**, AC-Q-4 |
| **`perf`** | `bench_index_scale` на референс-раннере → сравнить с зафиксированными числами `perf.md`, регресс > X% → fail | **AC-PERF-***, AC-PR-6 |
| **`coverage-full`** | полный отчёт + публикация тренда | AC-Q-2 (тренд) |
| `visual` (с Ф4) | Playwright `toHaveScreenshot` на собранном фронте | AC-PR-4 |

### Правило «no silent caps» (явный учёт пропущенного)

- **Rust:** прогон с `cargo nextest` (рекомендуется вместо голого `cargo test`: машиночитаемый вывод, ретраи для квантования флоки, явный список skipped). Шаг CI печатает и **проверяет число `#[ignore]`/skipped против ожидаемого whitelist** (`expected-ignored.txt`). Появился новый `#[ignore]`, не внесённый в whitelist → **CI падает** («tест молча отключён»). Так `live_eval_meets_baseline` и `bench_index_scale` — известные ignored (в whitelist), а случайный новый `#[ignore]` — нет.
- **Frontend:** Vitest с `--reporter=verbose`; шаг проверяет, что число `it.skip/describe.skip/it.todo` не выросло (или внесено в whitelist). `--allowOnly=false` в CI — `it.only` роняет прогон (защита от случайно закоммиченного `.only`, который «зеленит» прогон, скрывая остальное).
- **Traceability:** `pending`/`covered-live-only` AC печатаются отдельным блоком в лог CI с итоговым числом «N AC покрыто полностью, M — только live, K — pending». Зелёный CI при K>0 допустим (иначе не стартовать), но **число K на виду** в каждом прогоне.
- **Карантин флоки:** нестабильный тест помечается тегом (`#[ignore = "flaky: <issue>"]` / `it.skip` с комментом + запись в `traceability.yml` `status: quarantined`) и **выносится в отдельную non-blocking джобу** `flaky` (allow-failure), которая всё равно прогоняет их и репортит. Никогда не удаляем тихо — карантин виден.

## 6. Coverage

### Инструменты

- **Rust:** **`cargo-llvm-cov`** (LLVM source-based coverage; точнее tarpaulin, официально рекомендуемый, дружит с `nextest`: `cargo llvm-cov nextest`). Формат вывода — `lcov`/`json` для гейта и Codecov.
- **Frontend:** **`vitest --coverage`** с провайдером **v8** (`@vitest/coverage-v8`); добавить в `vitest.config.ts` блок `coverage` (сейчас его нет). Формат — `lcov`/`json-summary`.

### Стартовые пороги (из AC-Q-2, «старт 70%», калибруется)

- **Глобальные:** line ≥ 60% Rust, ≥ 60% frontend на старте (мягко — чтобы включить гейт без боли на текущей базе).
- **Критичные модули — выше, per-path порог:** `indexer`, `chunker`, `search` (RRF), `plugin/broker`, `plugin/permission`, `watcher`, `eval` (математика) — **цель ≥ 70%** (прямо как в AC-Q-2). **✅ реализовано** проверкой в скрипте `scripts/check-coverage.mjs` по JSON-отчёту `cargo-llvm-cov` (CI-джоба «Coverage (Rust)»; локально — `bash scripts/coverage.sh`): агрегирует покрытие строк по путям модулей и сверяет с floor'ами из `coverage-baseline.json`.
- **Факт первого замера (macOS):** chunker/permission ~98%, broker 93%, search 86%, indexer 70% — на/выше цели; **watcher 62% и eval 49% — ниже цели** → их floor зафиксирован по факту (ратчет «не регрессировать»), цель 70% растить отдельными тестовыми срезами. Floor'ы взяты с запасом под macOS↔Linux-вариативность; скрипт печатает факт. % каждый прогон → тюнить вверх по CI-Linux.

### Политика тренда (ratchet — «не падать ниже»)

- **Baseline в репозитории:** файл `coverage-baseline.json` с зафиксированным floor (глобально + per критичный модуль). **✅ есть** (Rust-слой).
- **Гейт PR:** покрытие в PR **не ниже** baseline (с допуском `tolerance` п.п. на macOS↔Linux/шум). Упало → **CI падает** с дельтой по модулям. **✅ реализовано** (`check-coverage.mjs`). Авто-bump baseline при росте + PR-комментарий с дельтой — **отложено** (см. `docs/BACKLOG.md`).
- **Только вверх:** при росте покрытия baseline **обновляется** (ручным шагом «bump coverage baseline» в release-прогоне или авто-PR-ботом), чтобы храповик не откатывался. Новый код без тестов физически роняет дельту → нельзя смержить непокрытую фичу. Это и есть механическая гарантия «тест на каждую новую функцию».
- **Codecov/артефакт:** lcov оба слоя публикуются (Codecov или GitHub-артефакт + PR-комментарий с дельтой), чтобы тренд был виден в PR.

## 7. Поэтапный план внедрения (оценки S/M/L), привязка к фазам бэклога

Порядок — по убыванию «ценности для автономности на единицу усилий». S ≈ часы, M ≈ день-два, L ≈ несколько дней.

| # | Шаг | Размер | Когда (фаза) | Почему здесь |
|---|---|---|---|---|
| 1 | **Единая команда «прогнать всё»** (`scripts/test-all.sh` + npm-скрипт `test:all`): `cargo nextest` + `vitest run` + опц. `--live` (eval/perf с сервером). Точка входа для агента перед PR. | **S** | сейчас (Ф1/Ф2 граница) | Разблокирует автономность немедленно, нулевая зависимость от нового кода. |
| 2 | **Coverage-гейт**: `cargo-llvm-cov` + `vitest --coverage` (блок в `vitest.config.ts`) + `coverage-baseline.json` + джоба `coverage` в `ci.yml` с ratchet. | **M** | Ф2 | Закрывает AC-Q-2 (сейчас декларация без измерения); сразу включает «не ниже» для всего нового кода. |
| 3 | **Traceability-док**: `traceability.yml` (заполнить по существующим 116+85 тестам и AC), `scripts/check-traceability.mjs`, джоба `traceability`, рендер `traceability.md`. Установить конвенцию `Closes AC-…` в шаблоне PR. | **M** | Ф2 | Делает «тест на каждую фичу» проверяемым; вскрывает, какие AC реально `pending`. |

Кэш-фикстура — предпочтительнее (детерминизм, без сервера в CI). |
| 7 | **Rust integration-крейт**: вынести/нарастить `src-tauri/tests/*.rs` (Tauri-команды end-to-end на tempfile-vault, миграции, git2, usearch, конкуренция SQLITE_BUSY). | **M** | Ф3 | Закрывает AC-Б7/Б8/Б9 как явный слой; дешевле E2E, ловит backend-регрессии кроссплатформенно (вкл. macOS). |

## 8. Как это усиливает АВТОНОМНОСТЬ агентской разработки

Ключевая ценность — агент сам гоняет полный набор и ловит регрессии **до** PR, а CI лишь подтверждает. Конкретные механизмы:

1. **Единая команда «прогнать всё» (`test:all`, шаг 1).** Агент не должен помнить «cargo здесь, vitest там, eval с сервером». Один вход: быстрый набор по умолчанию (детерминированный, минуты) + флаг `--live` для eval/perf. Это превращает «прогнать тесты» в одну атомарную операцию, которую агент выполняет рефлекторно перед каждым PR — как `git status`.

2. **Детерминизм (моки сети/LLM, фикстуры vault).** Автономность ломается о флоки и сетевые зависимости. Поэтому:
   - **Быстрый эшелон без живых серверов**: chat/embedding-вызовы в unit/integration замоканы (HTTP-эмбеддер тестируется на стабе, не на :8083). У нас уже есть **`lib/mock/*`** — детерминированный preview-backend (mock vault/git/broker/search/streamChat); контракт-тесты (§3, слой 2) держат его изоморфным реальным командам, так что агент может полностью разрабатывать и тестировать фронт **без поднятого Rust и без сети**.
   - **Фикстуры vault**: `tempfile`-vault (Rust, уже используется) и синтетический генератор (`bench_index_scale`) дают воспроизводимое окружение без реальных пользовательских данных. Для eval — **кэш-фикстура эмбеддингов golden** (шаг 6) делает RAG-гейт детерминированным и серверонезависимым в CI.
   - Недетерминированную часть (ранжирование) меряем **статистически против baseline** (eval), а не «угадал/не угадал» — это правильный способ тестировать LLM-зависимое поведение.

3. **Отлов регрессий ДО PR, механически.**
   - **Coverage-ratchet (шаг 2):** агент физически не может смержить новую функцию без теста — непокрытый код роняет дельту покрытия, PR краснеет. Это превращает «тест на каждую функцию» из дисциплины в инвариант сборки.
   - **Traceability-гейт (шаг 3):** новый `AC-…` без записи о тесте → красный CI. Агент, реализуя фичу, **обязан** связать её с тестом в матрице. «Каждая фича покрыта» становится проверяемым свойством, а не обещанием.
   - **Eval/perf регресс-гейты:** любое изменение чанкера/RRF/порогов/модели прогоняет eval (AC-EVAL-3); просадка ниже baseline блокирует merge. Perf-регресс >X% — то же. Агент узнаёт о деградации качества/скорости **числом**, а не от пользователя через неделю.
   - **No-silent-caps (шаг 4):** агент не может «починить красный CI», тихо добавив `#[ignore]`/`it.skip` — whitelist это поймает. Самый коварный анти-паттерн автономного агента (заглушить тест вместо фикса бага) закрыт механически.

4. **Двухэшелонный CI под скорость итерации.** Быстрый набор (минуты, на каждый PR) даёт агенту короткую петлю обратной связи; тяжёлый (E2E/eval-live/perf, на pre-release) ловит дорогие классы багов, не замедляя каждую итерацию. Агент итерирует на быстром, релиз гейтится полным — обе цели владельца («ловить при разработке» и «прогон при релизе фичи») выполнены раздельными, не конфликтующими эшелонами.

Итог: после шагов 1–4 у агента есть **одна команда** для полного локального прогона, **храповик покрытия** и **traceability-гейт**, которые делают «тест на каждую фичу/функцию» нарушаемым только через красный CI, и **защита от тихого заглушения тестов**. E2E (5) и CI-eval/perf (6) добавляют верхний слой пирамиды и автогейт качества/скорости. Это и есть инфраструктура, в которой регрессии ловятся без человека, а человек подключается только к содержательному ревью.
