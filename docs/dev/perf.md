# Производительность и нагрузка (AC-PERF, AC-DOD-Ф3)

> Что структурно сделано для масштаба 50k и **реальные числа** с нагрузочного бенчмарка
> полного пайплайна (индексация с живыми эмбеддингами).

## Структурный фундамент (Фазы 0–1, уже в main)

- **Дерево файлов** — виртуализировано (`@tanstack/react-virtual` + `flattenVisible`), дети грузятся
  лениво (`list_dir` по одному каталогу). Рендерится только видимое окно → масштабируется на 50k.
- **Граф** — раскладка в **Web Worker** (`layout.worker.ts`), main-thread не блокируется (AC-PERF-6).
  Единый граф ограничен топ-N по связности (по умолчанию 2000) — осмысленный обзор без перегруза.
- **Индексация** — инкрементальная (mtime/hash-reconcile, резюмируемая), идёт **фоном** → старт UI её
  не ждёт. Эмбеддинги — через HTTP-эмбеддер (bge-m3).
- **БД** — WAL + пул читателей; запись через единый write-actor (ADR-003).

## Нагрузочный бенчмарк (полный пайплайн, с эмбеддингами)

Харнесс: `eval::tests::bench_index_scale` (`#[ignore]`, не идёт в CI). Генерирует синтетический vault из
`NEXUS_BENCH_FILES` заметок (RU+EN тело + 3 вики-ссылки на соседей), индексирует **с живым bge-m3 :8083**,
замеряет индексацию / поиск / граф и проецирует время на 50k.

```sh
NEXUS_BENCH_FILES=2000 cargo test bench_index_scale -- --ignored --nocapture
```

### Результаты (bge-m3 :8083, локальный мак, ~1 чанк/файл)

| N | Скан | Индексация | emb/s | Поиск (гибрид) | Граф единый | Граф локальный |
|---|---|---|---|---|---|---|
| 500 | последовательный | 11.3 с | 44 | 105 мс | 3 мс (500 узлов) | <1 мс |
| 2000 | последовательный | 51.2 с | 39 | 21 мс | 18 мс (2000 узлов / 6000 рёбер) | <1 мс |
| 2000 | **конкурентный (Ф3-10)** | **35.9 с** | **56** | 20 мс | 18 мс | <1 мс |

### Выводы

- **Граф и поиск — интерактивны на масштабе.** Единый граф на пределе кэпа (2000 узлов / 6000 рёбер)
  строится за ~18 мс (плюс раскладка в Worker, вне main-thread); гибридный поиск 20–100 мс.
  **AC-PERF-6** и отзывчивость поиска подтверждены замером.
- **Старт UI не блокируется индексацией** (AC-PERF-1): индекс идёт фоном, открыть/искать/смотреть граф
  можно сразу — индекс «догоняет».
- **Конкурентный скан (Ф3-10): ~1.4×** (39 → 56 эмб/с, 50k ≈ 21 → 15 мин). `scan_vault` держит до
  `SCAN_CONCURRENCY` файлов «в полёте» (`buffer_unordered`) под семафором `EMBED_CONCURRENCY`. Это
  починило латентную неэффективность: раньше скан был **последовательным** и семафор конкуренции при
  первичном наполнении **простаивал** (1 запрос в полёте). Кооперативно в одной задаче → синхронные
  секции usearch/БД без гонок (без блокировок).
- **Потолок — инференс эмбеддера, а не клиент.** Прирост скромный (1.4×, не ×N), потому что локальный
  bge-m3 считает запросы ~последовательно (инференс-bound ~50 эмб/с на одиночных входах); конкуренция
  перекрывает лишь сетевой фрейминг/очередь. На быстром/удалённом/GPU-сервере конкуренция даст больше.
- Синтетика — ~1 чанк/файл (худший случай для батчинга): `embed_documents` шлёт батч до
  [`EMBED_BATCH`]=64 входов, но **в пределах одного файла** — на мелких файлах батч не наполняется.

## Что осталось (BACKLOG)

- **Cross-file батчинг эмбеддинга** — главный оставшийся рычаг для «многих мелких файлов»: копить чанки
  ПОПЕРЁК файлов в запросы по 64 → серверу один батч-форвард вместо N (инференс-batch кратно дешевле
  per-item). Инвазивнее (разнести embed-фазу и write-фазу скана).
- Прямой замер холодного старта фронта (<1 с) — индекс уже фоновый, но cold-load UI отдельно не мерян.
